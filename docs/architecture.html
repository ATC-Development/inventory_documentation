02

Component Responsibilities

inventory_pwa
React PWA Frontend
  • Login via MSAL; adapts UI for mobile and desktop
  • Captures barcode scans; displays inventory and reports
  • Sends access tokens to API and Functions on every request
  • Connects to SignalR hub for real-time cache invalidation (additive, non-blocking)
  • For report exports: sends intent only (filters, IDs, format) — never raw row data
  • Handles binary blob responses from Functions for file downloads
inventory_api
ASP.NET Backend
  • .NET 10 Minimal APIs + EF Core; SQLite in dev, Azure SQL in prod
  • Validates JWTs; enforces all authorization policies
  • Manages inventory with unique constraints throughout
  • Identifier resolution and enrichment workflow
  • Immutable transaction ledger for every quantity change
  • SignalR hub at /hubs/inventory; fire-and-forget broadcasts
  • Enqueues notification messages to storage queue (non-blocking)
  • Graph OBO quote send; Swagger UI in Development
inventory_functions
Azure Functions
  • .NET 9 Isolated Process; Flex Consumption, Linux
  • HTTP-triggered: reports, labels, invoice analysis — called by PWA
  • Timer-triggered: daily/weekly scheduled jobs
  • Queue-triggered: notification delivery via NotificationQueueFunction
  • Read-only DB access via Managed Identity; no direct DB writes
  • Calls API endpoints for write-back operations
  • Teams notifications via Power Automate flows
  • PdfSharpCore (PDF) · ClosedXML (Excel) · ZXing.Net (barcodes)

Identity & Authorization Flow

Identity is derived entirely from Entra ID JWT token claims. There is no local users table.

1 →User signs in via MSAL
2 →Access token includes oid, preferred_username, and roles
3 →PWA calls API with Authorization: Bearer <token>
4 →API validates token and enforces policies based on roles claim
5 →Audit fields written on every write: *_by_oid (required), *_by_email (optional)
RoleAccess
DatabaseManagerFull access — all properties, all admin and bulk operations
propCode_wh (e.g. ms_wh)Property staff — read all, write only to their assigned warehouse
No roleRead-only access to inventory data
ℹ️
DatabaseManager overrides all property role checks. A DatabaseManager can write to any warehouse regardless of property group membership. Role assignments are managed in Azure Portal → Enterprise Applications → ATC_InventoryAPI.

Functions Auth Extension

When the PWA calls an HTTP-triggered Function:

1 →PWA sends the same Bearer token used for API calls (access_as_user scope, ATC_InventoryAPI audience)
2 →JwtValidationMiddleware validates: signature, issuer, audience, expiry
3 →Middleware extracts oid and preferred_username from token claims
4 →Middleware cross-validates token oid against requested_by_oid in request body — mismatch returns 403
5 →Function accesses DB using Managed Identity — user token is never passed downstream
6 →Timer-triggered functions bypass JWT middleware — no HTTP context

Real-Time (SignalR)

Azure SignalR Service provides the managed WebSocket relay between the API and connected PWA clients. Hub endpoint: /hubs/inventory

Group Model

Client TypeGroup MembershipMechanism
Standard authenticated userSingle warehouse groupClient calls JoinWarehouse(id) on connect and on warehouse selection change
DatabaseManagerAll warehouse groupsHub adds to every active warehouse group in OnConnectedAsync
Enrichment notificationsNo group — user-targetedClients.User(oid) using OID claim from token
⚠️
All broadcasts are fire-and-forget. The API sends the event after the response is ready and never awaits client delivery. The PWA must degrade gracefully if the SignalR connection is unavailable — no feature may fail or hang due to a missing hub connection.
ℹ️
SignalR events are used exclusively for React Query cache invalidation — no event payload drives UI state directly. The PWA always re-fetches authoritative state from the API.

Barcode Scan Flow

Normal scan operations go directly to their dedicated endpoints. /identifiers/resolve is only triggered on 404 IDENTIFIER_NOT_FOUND — it is not part of normal scan operations.

Lane 1
Scan Out (Consume)
POST /supplies/scan/consume
  • Multiplier ignored — decrements by qty_each EA directly
  • Supply row must exist (allowCreate = false)
  • 404 → pivot to Lane 3
  • 409 → part not stocked at this warehouse
  • 400 → negative inventory blocked
Lane 2
Scan In (Stock-In)
POST /supplies/scan/stockin
  • Multiplier applied: qty_packages × Unit_multiplier_to_base = qty_change
  • Supply row auto-created if missing (allowCreate = true)
  • Auto-create triggers "New Part in Warehouse" transaction → Functions alert DatabaseManagers
  • 404 → pivot to Lane 3
Lane 3
Unknown Barcode
GET /identifiers/resolve/{code}
  • Returns Lookup_Started + lookup_id
  • Poll GET /identifiers/lookup/{id} every 1–2s
  • Complete → pre-fill part form → POST /parts
  • Exhausted / TimedOut → manual entry → POST /parts
  • After part created: stock-in or adjust to set initial stock
ℹ️
Multiplier example: Toilet Wax Ring 3-Pack UPC, Unit_multiplier_to_base = 3. User scans 2 packages: qty_packages = 2, qty_change = 2 × 3 = 6 EA. Supplies table stores +6. The multiplier is a stock-in conversion only — never persisted to the supply quantity.

06

Report Export Flow

Report generation is handled by Azure Functions, not the ASP.NET API. The PWA sends intent (filters, IDs, format) — never raw row data.

1 →User clicks export. Button disables and shows loading state. UI remains fully interactive.
2 →PWA acquires Bearer token (same access_as_user scope already in use)
3 →PWA POSTs to VITE_FUNCTIONS_BASE_URL/api/reports/{endpoint} with filters and report_meta
4 →Function validates token, fetches data from DB via Managed Identity, generates document
5 →Function returns binary file response (Content-Disposition: attachment)
6 →PWA calls response.blob(), creates object URL, triggers browser download. Button re-enables.
7 →If non-2xx, PWA reads JSON error body (check response.ok before calling response.blob())
ℹ️
Cold start latency on Consumption plan (2–4 seconds) is absorbed by the non-blocking UX pattern. Report generation is not on the scan-submit path.
07

Design Principles

API is the single source of truth — Functions and PWA must not bypass it for writes
Automation logic never blocks user requests — long work runs in Functions or background tasks
Queue-based notification delivery must not block user-facing API requests — written fire-and-forget
Identity comes from Entra ID, not local tables
Barcode/identifier uniqueness prevents duplicate parts; names are normalized and similarity-checked
Inventory changes append to a transactions ledger; negative inventory is blocked by the API
Supplies always store EA-level quantities; Unit_multiplier_to_base is applied at stock-in scan time only — never persisted
SignalR events are invalidation signals only — the PWA always fetches authoritative state from the API
Functions are read-only DB consumers — all writes go through the API
The PWA never writes to Azure Storage Queue — queue messages are produced by the API only
Secrets never in source control — Key Vault for all sensitive configuration across all three repos
Equipment is a separate domain from consumable inventory — location vs. quantity semantics
Quote request tracking is orthogonal to the order status lifecycle

Equipment Tracking

Equipment items (tools, large assets) are a separate domain from consumable inventory. They are not modeled as an extension of Parts/Supplies — the domains have fundamentally different semantics (location vs. quantity).

Architecture Pattern

TableRole
EquipmentStatusesCurrent truth — one row per item, upserted on every movement
EquipmentTransactionsImmutable ledger — append-only, full movement history

Barcode Namespace

Equipment barcodes start at sequence 8000001. Scan routing is determined by the scanned value:

  • Value ≥ 8000000 (7-digit numeric) → resolves against EquipmentIdentifiers
  • Value < 8000000 → resolves against Identifiers (consumable parts)
⚠️
The ATC_MASTER sequence is hard-capped at 7999999 (API-enforced) to prevent barcode namespace overlap with equipment identifiers.

Invoice Scan

Allows users to photograph or upload a vendor invoice. The system extracts line items via Azure Document Intelligence, matches them to known parts via a learned alias table and GPT-4.1-mini semantic matching, presents a verification modal, then writes stock-in transactions through the existing API.

Processing Pipeline

1 →Upload (multipart/form-data) → AnalyzeInvoiceFunction — JWT validated, propCode_wh role required
2 →ADI prebuilt-invoice model → extracts vendor name, line items, quantities
3 →VendorAliases lookup → instant resolution if alias exists (no AI call)
4 →GPT-4.1-mini matching for unresolved items → confidence-scored suggestions
5 →PWA InvoiceVerificationModal — user reviews all rows, edits matches
6 →On Confirm: POST /supplies/scan/stockin (batch) → all writes through API
7 →POST /vendor-aliases → persist confirmed mappings (alias learning loop)
✅
Human verification is mandatory. The Function never writes to the database — all writes occur via the API after user confirmation. There is no auto-commit path.

Quote Request — Graph OBO Send (F-10)

Allows users to send structured vendor quote request emails directly from the application. The API sends the email on behalf of the logged-in user via Microsoft Graph using the On-Behalf-Of (OBO) flow.

Auth Flow

1 →PWA acquires API-scoped Bearer token (access_as_user → ATC_InventoryAPI)
2 →PWA calls POST /quote-requests/{token}/send with Bearer token + { to_email, cc_emails }
3 →API validates inbound token
4 →API performs OBO exchange using ATC_InventoryAPI client certificate from Key Vault → Graph-scoped token (Mail.Send)
5 →API calls Graph sendMail as the user
6 →On success: sets Graph_sent_dt, writes QuoteRequestEmailEvent, fires orders.updated SignalR event
⚠️
Critical: The PWA Bearer token audience is api://ATC_InventoryAPI/access_as_user. This token cannot be passed directly to Graph — the API must perform the OBO exchange. Do not skip this step.
