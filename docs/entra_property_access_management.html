01

Overview

The Inventory Application uses two Entra ID app registrations to control access. Property-level permissions are driven entirely by App Roles defined on the API registration and assigned via M365 groups in Enterprise Applications.

✅
No code changes required. All access changes are made in the Azure Portal. Granting or revoking access to any property or administrative function is purely a configuration task.
02

The Two App Registrations

AppPurpose
ATC_Inventory_PWA Client app — handles user login and token acquisition only. No roles defined here.
ATC_InventoryAPI Resource/API app — defines all App Roles. This is where access is controlled.
⚠️
All role management happens on ATC_InventoryAPI, not ATC_Inventory_PWA.

04

How It Works

When a user logs in, Microsoft Entra issues a JWT access token containing a roles array. The token is evaluated client-side (for UI rendering) and server-side (for enforcement) on every request. No extra API calls are made to determine permissions — everything is derived from the token at login.

→User logs in
→Entra checks group memberships → resolves App Role assignments
→Access token issued with roles claim:
  { "roles": ["DatabaseManager"] } ← full access
  { "roles": ["inventoryApp", "ms_wh", "aug_wh"] } ← trained property staff
  { "roles": ["inventoryApp"] } ← trained, no property role yet
  { "roles": [] } ← untrained / read only
→PWA reads roles → drives UI visibility and write gates
→API reads roles → enforces write permissions

06

Adding a New Property

When a new property comes online, complete these steps in order.

1
Create the App Role
Azure Portal → App Registrations → ATC_InventoryAPI → App Roles → Create App Role
Display name: Full property name (e.g. Riverside Commons)
Allowed member types: Users/Groups
Value: propCode_wh — no spaces, lowercase (e.g. rc_wh)
Description: Property staff access for Riverside Commons
Enabled: Yes
The Value field is critical — it must follow the propCode_wh convention exactly. This value appears in the JWT token and is matched against the warehouse record in the database.
2
Assign the M365 Group to the Role
Azure Portal → Enterprise Applications → ATC_InventoryAPI → Users and Groups → Add Assignment
1. Click Add user/group
2. Search for and select the property's M365 group (e.g. Riverside Commons)
3. Under Select a role, choose the newly created App Role (e.g. rc_wh)
4. Click Assign
3
Update the Warehouse Record
In the Inventory Application, a DatabaseManager must set the propCode on the new warehouse record to match the role convention (e.g. rc). The application derives the role as propCode + '_wh' automatically.
4
Verify
Have a property staff member log out and back into the PWA. Their token will contain the new role on next login.
07

Managing the DatabaseManager Role

DatabaseManager is a special App Role that grants complete access to all properties and all administrative functions.

Grant DatabaseManager Access

  1. Add the user to the Inventory-DatabaseManagers M365 group in Teams or Entra Admin
  2. The user must log out and back in for the token to reflect the change

Revoke DatabaseManager Access

  1. Remove the user from the Inventory-DatabaseManagers M365 group
  2. The user must log out and back in for the change to take effect
ℹ️
Inventory-DatabaseManagers group is assigned to the DatabaseManager App Role in Enterprise Applications → ATC_InventoryAPI → Users and Groups.
08

Managing the InvApp_Users Role

The inventoryApp App Role is the write-access gate for the Inventory Application. It must be assigned to any user who needs to perform write operations (scan out, stock-in, add parts, adjustments, equipment actions). DatabaseManagers bypass this gate automatically.

This role is assigned by adding users to the InvApp_Users M365 group, which is assigned to the inventoryApp App Role in Enterprise Applications.

Grant Write Access (After Training)

  1. Add the user to the InvApp_Users M365 group in Teams or Entra Admin
  2. The user must log out and back in for the token to reflect the change

Revoke Write Access

  1. Remove the user from the InvApp_Users M365 group
  2. The user must log out and back in for the change to take effect
ℹ️
InvApp_Users group is assigned to the inventoryApp App Role in Enterprise Applications → ATC_InventoryAPI → Users and Groups.
⚠️
New employees are read-only by default. Any authenticated ATC user can log into the PWA and browse inventory, but cannot perform any write operations until they are added to InvApp_Users after completing training. This is by design.
✅
DatabaseManagers do not need to be in InvApp_Users. The DatabaseManager role already implies full write access. Adding a DM to InvApp_Users is harmless but unnecessary.
09

Managing Property Access for Individual Users

Property access is controlled entirely by M365 group membership.

ActionSteps
Grant accessAdd user to the property's M365 group in Teams or Entra Admin
Revoke accessRemove user from the property's M365 group
Move to new propertyRemove from old group, add to new group
Multi-property accessAdd to multiple property groups
ℹ️
Changes take effect on the user's next login. Active sessions are not immediately affected — the user must log out and back in.
10

Token Behavior Notes

⚠️
Write access requires inventoryApp. A user with only property roles but without inventoryApp in their token will be treated as read-only. Both inventoryApp and a property role are required for property write access.
ℹ️
A user with no group assignments receives a token with no roles claim — the application treats this as read-only automatically.
ℹ️
A user assigned to multiple property groups will have all corresponding roles in their token (e.g. ["inventoryApp", "ms_wh", "aug_wh"]).
✅
DatabaseManager overrides all role checks — a DatabaseManager can write to any warehouse and bypasses the inventoryApp gate entirely.
⚠️
The 901 / ATC Office warehouse has no corresponding App Role by design. Only DatabaseManager users can perform write operations there.
