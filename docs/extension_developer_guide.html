01

Overview

The ATC Inventory Extension is a Manifest V3 Chrome/Edge extension that automates filling vendor web carts on behalf of property managers. It is the fourth component of the ATC Inventory system alongside inventory_api, inventory_pwa, and inventory_functions.

The extension bridges the PWA and the vendor's web platform. The PWA orchestrates the session lifecycle — login verification, preflight validation, order confirmation — while the extension handles the actual browser automation: detecting login state, clearing existing cart items, and bulk-posting order line items to the vendor's API.

ℹ️
First vendor: Fox Appliance Parts — foxparts.com, running the Kodaris B2B e-commerce platform. The architecture is intentionally generalized to support additional vendors with separate per-vendor content scripts and configuration.

What the extension does not do

The extension does not store credentials, manage sessions beyond the current fill operation, make API calls to the ATC inventory system, or hold any state beyond what is needed for the current active cart fill. All business logic, order status transitions, and audit trails live in inventory_api.

Key Constraint — Two Content Script Worlds

The Fox vendor integration uses two content scripts running in different browser execution worlds on foxparts.com:

FileWorldPurpose
content.main.js MAIN Reads kd global (Kodaris state object) which is not accessible from the isolated world. Passes csrfToken, isLoggedIn, selectedCompanyID to the isolated world via window.postMessage. Contains no fetch calls.
content.isolated.js ISOLATED Executes all fetch calls, cart operations, and overlay rendering. Receives identity values from the MAIN world. All network requests run here — the page's JavaScript cannot intercept them.
⚠️
Never put fetch calls in content.main.js. The MAIN world is accessible to the page's own JavaScript. All network calls that could expose credentials or CSRF tokens must run in the ISOLATED world only.
03

Architecture

Components

ComponentLocationRole
background.js Service Worker Message router and session manager. Handles all chrome.runtime.sendMessage calls. Owns one active session at a time (step: 'login' | 'fill'). Pushes async messages to the PWA tab via chrome.tabs.sendMessage.
content-bridge.js PWA tab (ISOLATED) Injected into the PWA origin. Bridges window.postMessage from the PWA page into chrome.runtime.sendMessage (outbound) and forwards push messages from background back into the PWA page via window.postMessage (inbound).
content.main.js Vendor tab (MAIN) Reads Kodaris kd global and relays identity values to the isolated world.
content.isolated.js Vendor tab (ISOLATED) Handles login detection, pending login tracking across page reloads, cart inspection, cart clearing, bulk item POST, post-fill cart verification, and the post-fill overlay.

Active Session Model

The background maintains a single activeSession object — only one cart fill operation can be in progress at a time. The session tracks: step (login or fill), vendorKey, correlationId, tabId (vendor tab), pwaTabId (PWA tab), and a timeout handle.

ℹ️
correlationId is generated by the PWA and passed through every message. The PWA validates it on all async push messages to prevent stale responses from a previous session from being processed.
04

Message Flow

Step 1 — Login Check

1 →PWA sends check_login via window.postMessage (with vendorKey, correlationId)
2 →content-bridge forwards to background via chrome.runtime.sendMessage
3 →background opens/focuses the vendor tab and sends check_login to content.isolated.js
4a →Logged in: isolated script returns login_verified → background sends immediate response to PWA
4b →Not logged in: isolated stores PENDING_LOGIN_KEY in session storage, shows login banner → returns login_required immediately
5 →On page reload after login: checkPendingLogin() detects login, clears flag, pushes login_ready to PWA asynchronously

Step 2 — Cart Fill

1 →PWA sends fill_cart with { vendor_key, items[], warehouse_name, correlationId }
2 →background opens/focuses vendor tab and sends fill_cart payload to content.isolated.js
3 →isolated script checks existing cart — if not empty, shows clear-cart warning overlay
4a →User cancels clear: returns { success: false, error: 'user_cancelled' }
4b →User confirms / cart empty: DELETEs existing items via Promise.all(), then bulk POSTs new items
5 →isolated verifies cart via GET, stores overlay state, navigates to cart page
6 →background pushes cart_filled (or cart_fill_failed) to PWA tab asynchronously

Security Model

PWA Origin Validation

The background service worker maintains an APPROVED_PWA_ORIGINS allowlist. Every message received from a content script is validated against this set before processing. Messages from unapproved origins are rejected with a 403-equivalent error.

const APPROVED_PWA_ORIGINS = new Set([
  'http://localhost:5173',
  'https://gray-tree-0fe80630f.4.azurestaticapps.net',
]);
⚠️
Update this set when the PWA URL changes. Adding a new Azure Static Web Apps deployment or a custom domain requires adding the new origin here and rebuilding.

content-bridge.js Filtering

The content-bridge only forwards messages that pass all of these checks:

  • event.source === window — same window only
  • event.origin === PWA_ORIGIN — same origin only
  • data.source === 'ATC_PWA' — correct source tag
  • ALLOWED_TYPES.has(data.type) — only ping, fill_cart, check_login

Sensitive Values

ValueHandling
Fox session cookiesHttpOnly — never accessible to JavaScript in any world
CSRF token (kd.csrfToken)Read in MAIN world, passed to ISOLATED via postMessage. Never logged or stored beyond the current fill operation.
callback_tokenCarried in PWA session state only. Never passed to the extension — the PWA calls cartConfirm directly using the stored token.
session_idSame as callback_token — stays in the PWA, never sent to the extension.

No Stored Credentials

The extension stores no vendor credentials. The manager logs in manually via the browser. The extension only reads identity state that the vendor's own page exposes after login (kd.customer.isLoggedIn, kd.customer.selectedCompanyID).

Fox Parts — Platform Details

Fox Appliance Parts runs on the Kodaris B2B e-commerce platform. All discoveries below were confirmed by live testing against an authenticated ATC session.

Identity Detection

Kodaris exposes a kd global object on every page via an inline script block. It is not attached to window and must be accessed directly. Two fields are used for identity verification:

kd.customer.isLoggedIn        // boolean
kd.customer.selectedCompanyID  // ATC's company ID: 2771
kd.csrfToken                   // Required as X-CSRF-TOKEN header on mutating requests
ℹ️
kd is only accessible from the MAIN world. The isolated world cannot read it directly. content.main.js reads it and passes the values to content.isolated.js via window.postMessage.

Confirmed API Endpoints

MethodEndpointPurpose
GET/api/user/cartRetrieve current cart state — orderID and line items
POST/api/user/cart/item/bulkAdd multiple items by part code and quantity
DELETE/api/user/cart/item/{orderItemID}Remove single item — no bulk delete endpoint exists

Cart Clearing

No clear-all cart endpoint exists. Per-item DELETE is required. All DELETEs are issued in parallel via Promise.all() before the bulk POST. Never chain them sequentially — the Fox API does not require ordering and parallel execution is significantly faster.

const deleteResponses = await Promise.all(
  existingItems.map((item) =>
    fetch(`/api/user/cart/item/${item.orderItemID}`, {
      method: 'DELETE',
      credentials: 'include',
      headers: { 'X-CSRF-TOKEN': csrfToken },
    })
  )
);

Authentication

All Fox API calls require two things: session cookies (sscaltoken and JSESSIONID — HttpOnly, set by Fox on login, not accessible to JavaScript) and the CSRF token set as X-CSRF-TOKEN on all mutating requests. The credentials: 'include' fetch option handles cookie forwarding automatically.

Adding a New Vendor

The extension is designed to support multiple vendors. Each vendor gets its own content scripts and configuration. The shared infrastructure (background.js, content-bridge.js, manifest) does not need to change for most additions.

1
Create vendor directory and config
Create src/vendors/{vendorkey}/ with a vendor.config.js file matching the Fox pattern. Set vendor_key to match the value that will be stored in Vendors.Vendor_key in the database.
2
Create content.main.js
Handles reading any page-level globals that are not accessible from the isolated world. If the vendor's platform does not use MAIN world globals, this file can be minimal — just respond to the request message and return empty values.
3
Create content.isolated.js
Implement handleCheckLogin() and handleFillCart(payload) following the Fox pattern. Every handleFillCart path must return either { success: true, ... } or { success: false, error: '...' } — no silent failures or unresolved promises. Use 'user_cancelled' specifically when the user cancels a warning dialog.
4
Register in background.js
Add the new vendor config to VENDOR_CONFIG: 
import { NEW_VENDOR_CONFIG } from './vendors/newvendor/vendor.config.js';

const VENDOR_CONFIG = {
  fox: FOX_CONFIG,
  newvendor: NEW_VENDOR_CONFIG,  // add here
};
5
Update manifest.json
Add the new vendor's domain to host_permissions and add two new content_scripts entries (MAIN and ISOLATED worlds) targeting the new domain.
6
Register vendor in build.js
Add the new vendor key to ALL_VENDORS in build.js: 
const ALL_VENDORS = ['fox', 'newvendor'];
7
Database — add vendor records
In the inventory database: add a Vendors row with Vendor_key matching your config, add a VendorOrderMethods row with method = 'web_cart' and a valid Config_json, and populate VendorParts cross-reference rows with the vendor's part codes.
8
Build and test
Run npm run build, reload the extension, and test the full flow from the PWA On Order page. Verify login detection, cart clear warning, fill, confirmation, and the post-fill overlay.
✅
No API changes required for new vendors — the session management, collision detection, confirm/mark-ordered flow, and all status transitions reuse existing endpoints. The only API change needed is the database records and potentially a web_cart Config_json validation rule for the new vendor's required fields.

Deployment

The extension is deployed to all managed Edge browsers via Microsoft Intune using Edge ADMX policy. Users do not install it manually.

Build for Deployment

bash — build and package
npm run build
# Zip the dist/fox/ directory contents (not the folder itself)
# The zip must contain manifest.json at the root level
⚠️
Update the version in manifest.json before each deployment. Intune and Edge use the version number to detect updates. Increment following semver: patch for bug fixes, minor for new vendor support, major for breaking changes to the message protocol.

Hosting the Extension Package

The packaged extension (.crx or .zip) must be hosted at a stable URL accessible to managed devices. GitHub Releases on the inventory_extension repo is the recommended approach — tag each release and use the release asset URL in the Intune policy.

Intune Deployment via Edge Policy

See the companion document Extension Intune Deployment for the step-by-step ADMX policy configuration. The key policy setting is ExtensionInstallForcelist which takes a value in the format:

extensionid;https://path/to/update_manifest.xml

An update manifest XML file must be hosted alongside the extension package to support automatic updates.

Update Manifest Format

<?xml version='1.0' encoding='UTF-8'?>
<gupdate xmlns='http://www.google.com/update2/response' protocol='2.0'>
  <app appid='YOUR_EXTENSION_ID'>
    <updatecheck
      codebase='https://your-host/atc-inventory-extension-1.0.0.crx'
      version='1.0.0' />
  </app>
</gupdate>
