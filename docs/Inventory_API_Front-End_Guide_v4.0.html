1. Authentication & Authorization

All endpoints require a valid Bearer token obtained from Microsoft Entra ID via MSAL. Include the token on every request:

Authorization: Bearer <access_token>
Content-Type: application/json

Authorization Roles

Three access tiers are enforced via the roles claim in the JWT. The API enforces these server-side on every request — the front end may use role claims for UI show/hide only and must never treat them as the source of truth.

RoleAccessHow Assigned
Authenticated User Any valid Entra ID token with no role assignment. Read all data. Single creates (parts, supplies, identifiers). Scan consume and stock-in. Part updates. Order create and status update. Equipment checkout/checkin/location update. Any valid Entra user.
propCode_wh (e.g. ms_wh) Read all. Write scoped to their assigned warehouse only. Warehouse identified by matching PropCode + _wh suffix. Can stock-in-receive orders for their warehouse. App Role per property, assigned in Azure Portal → Enterprise Applications → ATC_InventoryAPI. See Entra_Property_Access_Management.md.
DatabaseManager Full access — all warehouses, all admin and bulk operations. Overrides all property role checks. Automatically joined to all warehouse SignalR groups on connect. M365 group Inventory-DatabaseManagers → App Role in Azure Portal. No code change required.
403 Forbidden on a DatabaseManager-gated endpoint means the user is not in the M365 group — nothing is wrong with the application. Direct them to contact an administrator.
The PWA resolves roles at runtime by decoding the JWT in useUserPermissions(). isDatabaseManager and canWriteToWarehouse(propCode) are available throughout the app. These are UX signals only — the API always enforces independently.

Single-Create vs Bulk Split

This is a deliberate pattern. Single creates (parts, supplies, identifiers) are open to any authenticated user. Bulk and destructive operations require DatabaseManager. PUT /parts/{id} follows the single-create rule.

4. SignalR — Real-Time Events

Connect to the hub at /hubs/inventory using the same Bearer token used for API calls. All events are cache-invalidation signals only — the PWA re-fetches authoritative state from the API on receipt. Never drive UI state directly from event payloads.

The hub connection is additive and non-blocking. The application must function fully if the SignalR connection is unavailable. No feature may fail or hang due to a missing connection.

Event Catalogue

supply.updated
→ Group: warehouse_{id}

Any supply quantity change. Payload: { warehouseId, partId }

supply.low_stock
→ Group: warehouse_{id}

Supply write where quantity <= min_level. Payload: { warehouseId, partId, currentQty, minQty, partName, warehouseName }

part.created
→ All clients

Single part created or bulk batch complete. Payload: { partId } or { count }

part.updated
→ All clients

Part updated or deleted. Payload: { partId }

warehouses.changed
→ All clients

Warehouse create, update, or delete. No payload.

categories.changed
→ All clients

Category create, update, or delete. No payload.

vendors.changed
→ All clients

Vendor create, update, or delete. No payload.

orders.updated
→ All clients

Order status change, order created, stock-in-receive, or Graph OBO send accepted. No payload.

equipment.changed
→ All clients

Any equipment write — create, update, soft-delete, checkout, checkin, location update, permanent transfer. No payload.

Group Membership

Warehouse group membership controls who receives supply.updated and supply.low_stock events. Group name format: warehouse_{id}.

Client TypeGroup MembershipMechanism
Standard userSingle warehouse group — currently selected warehouseClient calls JoinWarehouse(id) on connect and on warehouse change; calls LeaveWarehouse(id) before switching
DatabaseManagerAll warehouse groupsAuto-added to every warehouse group in OnConnectedAsync — role checked server-side from token claim

POST /orders/{id}/stock-in-receive DM or propCode_wh

Atomic operation: adds received quantity to the supply row, writes a Transaction ledger row (ReasonType = ORDER_RECEIPT, ReferenceId = order.Id), and marks the order received — all in a single database commit.

Supply row must already exist for this part+warehouse. Returns 409 SUPPLY_NOT_FOUND if not. Use normal scan/stockin to create the initial supply row before receiving against it.
FieldTypeRequiredNotes

// Response: 200 OK — updated InventoryOrderReadDto
// Broadcasts: supply.updated (warehouse group) + orders.updated (all clients)
// Error: 409 INVALID_TRANSITION — order already received or cancelled
// Error: 409 SUPPLY_NOT_FOUND — supply row does not exist for this part+warehouse
// Error: 409 RECEIPT_CONFLICT — concurrent write conflict, retry

15. Equipment NEW

Tracks discrete, physically identifiable tools and equipment. Separate domain from consumable inventory — equipment has a location at all times rather than a quantity.

Barcode namespace: Equipment barcodes are purely numeric starting at 8000001. Any scanned value ≥ 8000000 (7-digit numeric) resolves against /equipment/scan/resolve. 17. Barcode Scanning Workflows

/identifiers/resolve is NOT part of normal scan operations. Call it only when a scan returns 404 IDENTIFIER_NOT_FOUND.
The Supplies table ALWAYS stores EA (base unit) quantities. The multiplier conversion happens at stock-in scan time and is never persisted.

Lane 1 — Scan Out (Consume)

POST /supplies/scan/consume — single call, no resolve step.

Multiplier is ignored. Decrements by qty_each EA. Supply row must exist.

ErrorMeaningAction
404 IDENTIFIER_NOT_FOUNDBarcode unknownPivot to Lane 3 (unknown barcode)
409 SUPPLY_NOT_FOUNDPart not stocked at this warehouseShow error — cannot consume what isn't here
400 NEGATIVE_INVENTORY_NOT_ALLOWEDWould go below 0Show current quantity and block action

Lane 2 — Scan In (Stock-In)

POST /supplies/scan/stockin — single call, no resolve step.

Multiplier applied: qty_packages × Unit_multiplier_to_base = qty_change. Supply row auto-created if missing.

Example: Toilet Wax Ring 3-Pack, multiplier = 3. Scan 2 packages → qty_packages = 2, qty_change = 6 EA.

ErrorMeaningAction
404 IDENTIFIER_NOT_FOUNDBarcode unknownPivot to Lane 3 (unknown barcode)
400 INVALID_UNIT_MULTIPLIERMultiplier ≤ 0 — data errorReport to admin

Lane 3 — Unknown Barcode

Triggered when Lane 1 or 2 returns 404 IDENTIFIER_NOT_FOUND.

  1. Call GET /identifiers/resolve/{code} → returns Lookup_Started + lookup_id
  2. Poll GET /identifiers/lookup/{lookupId} every 1–2 seconds. No auth required.
  3. Stop polling when status ≠ Pending.
  4. Complete → pre-fill part creation form with enriched data → POST /parts
  5. Exhausted / TimedOut → manual entry fallback → POST /parts
  6. After part created → POST /supplies/scan/stockin or POST /supplies/adjust for initial stock.

Equipment Scan (Two-Phase)

Equipment barcodes (≥ 8000000, 7-digit numeric) use a separate two-phase flow.

  1. Call POST /equipment/scan/resolve with { "code": "8000011" }
  2. Response includes equipment (EquipmentReadDto) and available_actions array.
  3. Present only the actions listed in available_actions to the user.
  4. Call the appropriate action endpoint: checkout, checkin, or locationupdate.
Mobile shorthand: On the equipment scan screen only, numeric-only input is treated as a TOOL_MASTER suffix — the PWA prepends 8 and zero-pads to 7 digits. User types 11 → resolves as 8000011. This shorthand is only active when the equipment scan screen is open.

18. General Error Handling

All errors return a JSON body. Standard shape:

// Standard error
{ "error": "ERROR_CODE", "message": "Human-readable description." }

// With suggestions (name similarity on POST /parts)
{ "error": "NAME_TOO_SIMILAR", "message": "...", "suggestions": [{ "id": 7, "name": "..." }] }

// Identifier conflict (includes existing part for UI resolution)
{ "error": "IDENTIFIER_CONFLICT", "message": "...",
  "existing_part": { "id": 7, "name": "1/2 inch Ball Valve" },
  "existing_identifier": { "id_type": "UPC", "id_value": "012345678901" } }

// Status transition conflict
{ "error": "INVALID_TRANSITION", "current_status": "received", "allowed_transitions": [] }
