Purpose of This Guide

This guide is the primary developer reference for the ATC Inventory PWA. It is written for developers who need to maintain, troubleshoot, extend, or deploy the React frontend without relying on project history or informal knowledge.

The PWA is the user-facing application for inventory operations. It owns the user experience, Microsoft Entra sign-in, API access-token acquisition, desktop and mobile workflows, React Query cache management, SignalR client integration, scanner UI, and PWA update behavior.

⚠️
The PWA is not the system of record. It does not enforce final authorization, does not write directly to the database, does not own inventory business rules, and does not host scheduled automation. Those responsibilities remain in the API and Functions repositories.
ℹ️
This guide covers: local setup, build, MSAL auth, role-based UI, warehouse context, TanStack Query, SignalR, PWA service worker, dual desktop/mobile architecture, scanner workflow, page families, testing, troubleshooting, and change impact procedures.

This guide does not cover: full API endpoint contracts (API_CONTRACT.html), API business rule implementation (API_DEVELOPER_GUIDE.html), Azure Functions internals (FUNCTIONS_DEVELOPER_GUIDE.html), or end-user task instructions (public/guide/index.html).

Source-of-Truth Rules

1Current PWA source code wins for implemented frontend behavior. The snapshot is the authority for routing, provider placement, component behavior, query keys, auth bootstrapping, service worker configuration, and implemented UI patterns.
2The API contract wins for endpoint shape. Do not duplicate API request and response contracts in this guide. Reference API_CONTRACT.html.
3The API source code wins for server-side behavior. The PWA may guide the user, but the API validates tokens, enforces authorization, applies inventory rules, and protects data integrity.
4Architecture and project rules win for repository boundaries. The PWA, API, and Functions repositories must remain independently deployed systems.
5When uncertain, flag uncertainty. Do not document guessed behavior as implemented behavior.

Build & Configuration

Vite Configuration

The Vite configuration owns: React plugin setup, Tailwind integration, PWA plugin configuration, alias configuration (@ to src), build-time version injection, and static asset/service worker behavior.

Before changing Vite configuration, test: local dev server startup, production build, Static Web App routing fallback, PWA installation and update prompt, MSAL login redirect, and service worker update behavior.

Version Injection

Use semantic versioning; version visible in app must match the deployed artifact
public/version.json must remain network-fresh — do not let the service worker precache stale version metadata

TypeScript and ESLint

Treat type errors and lint errors as build-quality issues, not optional cleanup. Prefer explicit types for API payloads and UI state. Keep DTO-like frontend types aligned to API_CONTRACT.html. Avoid duplicating backend validation rules unless the UI needs a friendly pre-submit guard.

Static Web App Configuration

staticwebapp.config.json controls SPA routing fallback and HTTP headers. Expected concerns: SPA fallback to index.html, service worker and Workbox files not over-cached, version.json network-fresh, and security headers must not break MSAL redirect or iframe-based token flows.

⚠️
Verification note: During final review, verify the exact staticwebapp.config.json that Azure Static Web Apps consumes. If multiple config files exist, document which one is active and reconcile the others.

PWA & Service Worker Behavior

Update Strategy

The app uses a prompt-based update flow. Users are notified that an update is available and can choose when to reload.

⚠️
Do not switch to automatic reload. A forced refresh during a scan, receive, order, or edit modal can cause user confusion or lost UI state.

Service Worker Caching Rules

Resource TypeIntended Behavior
App shell and static assetsCan be cached for faster startup and offline shell availability
version.jsonMust be network-fresh; do not precache stale version metadata
Microsoft identity endpointsMust not be intercepted by app caching logic
API requestsMay use network-first caching only when implemented; writes require network
User guide static assetsCan tolerate longer caching than auth/version metadata

Offline Capability and Limits

Authenticated writes require network access — scan in, scan out, receive order, edit, delete, and create actions require API availability
SignalR is unavailable offline and must not block normal app use
Cached data may be stale — the user must receive clear failure feedback when a network-only operation cannot complete

Version Check

useVersionCheck and UpdatePrompt detect a newer deployed build and prompt the user to reload. Fetch version.json from the network, compare deployed version to current app version, show a non-blocking prompt, allow the user to continue current work, and reload only when the user chooses to apply the update.

Authentication

MSAL Configuration

msalConfig.ts defines the MSAL browser configuration: Entra client ID and tenant authority, redirect and post-logout redirect behavior, cache behavior, login scopes, and API access scopes.

RequestPurpose
Login requestBasic identity scopes: openid, profile, email, and optional Microsoft profile scopes
API requestAccess token for the Inventory API scope, usually access_as_user

Bootstrap Order

The application must initialize MSAL and consume redirect responses before rendering protected app routes. Rendering authenticated routes before MSAL finishes redirect handling causes route thrashing, stale account state, or redirect loops.

order
create PublicClientApplication singleton
→ msalInstance.initialize()
→ msalInstance.handleRedirectPromise()
→ render React app

AuthShell Hardening and Redirect-Loop Fix

AuthShell is the guard between unauthenticated routes and the authenticated application.

Wait while MSAL interaction is in progress
Never initiate a new redirect while inProgress !== "none"
Render a safe transitional state while auth is settling
Keep AuthenticatedShell separate from AuthShell — authenticated-only providers must mount after login
Prevent stale MSAL interaction state from trapping the user in a loop

Auth Troubleshooting Quick Reference

SymptomLikely Area

Token Acquisition for API Calls

1 →Use MSAL account context
2 →Request API scope with acquireTokenSilent
3 →Attach Authorization: Bearer <access_token>

5 →Parse API errors into a user-appropriate message; let route/page code handle forbidden or failed actions gracefully

Warehouse Context

Warehouse context is one of the core app-level states in the PWA. It affects search, scan, inventory lists, reports, orders, and SignalR group membership.

Responsibilities

Warehouse context provides: available warehouses, selected warehouse, all-properties mode where supported, manual selection behavior, location-aware nearest warehouse behavior where implemented, loading and error states, and helper methods for consumers.

All-properties mode is not a valid mutation target. All write actions must resolve to a specific warehouse.
Mobile workflows must keep property context visible — a user scanning in the wrong property can create real inventory errors
When warehouse selection changes, the SignalR client must join/leave the appropriate warehouse group

Data Fetching: TanStack Query v5

TanStack Query v5 handles server state. Query keys and invalidation are cross-cutting infrastructure — treat them as architecture, not incidental implementation.

Query Client Defaults

  • Operational data defaults to finite stale time, commonly 5 minutes
  • Retry count is low, commonly 1
  • Reference data may override stale time to Infinity
  • SignalR and explicit mutation handlers invalidate stale caches

Query Key Factory

queryKeys.ts is the central query key factory.

Use the factory for normal query keys — avoid ad hoc string arrays for new feature queries
Keep query key shape stable — changing it can silently break invalidation
Add new keys to the factory before using them broadly

Reference Data vs. Operational Data

Reference Data
staleTime: Infinity
  • Warehouses
  • Categories
  • Vendors
  • Refresh via: mutation invalidation, SignalR changed events, manual refresh
Operational Data
staleTime: 5 minutes
  • Parts, Supplies, Transactions
  • Orders, Equipment, Reports
  • Invoice results
  • Refresh via: writes, relevant SignalR events

Mutation Invalidation Patterns

MutationExpected Invalidation
Scan in / stock insupplies, transactions, dashboard/report data, related orders where applicable
Scan out / consumesupplies, transactions, usage/report data
Part create/updateparts, identifiers, vendor cross-reference data, supplies if warehouse assignment changed
Identifier add/relinkpart detail, identifier lookup/cache, scan lookup assumptions
Warehouse create/update/deletewarehouses reference data and warehouse-scoped views
Category create/update/deletecategories reference data and part/catalog views
Vendor create/update/deletevendors reference data and vendor cross-reference views
Order updateorders, order history, supplies if stock-in receive occurred
Equipment movementequipment list/detail, equipment transactions
⚠️
Do not store server truth in local UI state. Local component state can hold form values, selected rows, modal mode, and temporary UI flags. After writes, rely on invalidation/refetch patterns unless a deliberate optimistic update is implemented and tested.

SignalR Integration

Architecture Rules

SignalR connection should be app-level, not page-level — SignalRProvider lives high in the authenticated app shell
SignalR must degrade gracefully — normal API operations must still work if SignalR is disconnected
Events should not be treated as authoritative data payloads — they are invalidation signals
Hub path is /hubs/inventory — do not use the older /inventoryhub reference

Event Handling Pattern

✓Correct: SignalR event received → determine affected query namespace → invalidate TanStack Query keys → UI refreshes from API data
✗Avoid: SignalR event received → directly mutate visible UI as source of truth

Toast notifications may use event payloads for display-only messages when explicitly implemented, such as low-stock notifications via supply.low_stock.

Connection Lifecycle

When reconnecting, the client must regain any required warehouse group membership. If access tokens expire, reconnect logic must obtain a fresh token.

SignalR Failure Behavior

Show connection status only if useful — do not block page rendering
Do not block scan or write operations due to SignalR unavailability
Continue using API calls and mutation invalidation; allow reconnect attempts in the background

Scan Workflow

Scanner Architecture

Native browser BarcodeDetector path when supported
Quagga2 fallback when native detection is unavailable
Manual barcode entry fallback in all cases — never remove manual fallback

Barcode Normalization

Barcode normalization must be centralized. Expected responsibilities: trim and normalize raw scan/manual input, normalize ATC shorthand where implemented, normalize case for identifier comparison, apply numeric padding rules, and distinguish supply/part barcode behavior from equipment/tool barcode behavior.

⚠️
Any change to normalization can affect: scan in, scan out, add identifier, relink identifier, part lookup, equipment scan, and vendor/invoice workflows. Avoid page-specific one-off normalization.

Scan Out / Consume Flow

1 →Confirm selected warehouse
2 →Scan or enter barcode
3 →Resolve part/supply using cache or API lookup path
4 →Confirm matched part and quantity
5 →POST consume request to API — API enforces business rules including negative inventory protection
6 →Show success or error; invalidate affected queries

Scan In / Stock-In Flow

1 →Confirm selected warehouse
2 →Scan or enter barcode; resolve part/supply
3 →Confirm matched part and quantity
4 →POST stock-in request to API — API applies identifier unit multiplier rules where applicable
5 →Show success or error; invalidate affected queries

Unknown Barcode / Identifier Resolution Flow

1 →Scan or enter code → normal scan lookup fails with identifier-not-found condition
2 →Call identifier resolve endpoint only for unknown identifier flow
3 →API returns FOUND or lookup started
4 →If lookup started, poll lookup status
5 →If complete with useful metadata, prefill new-part flow; otherwise prompt manual entry
ℹ️
Normal scan operations should not call identifier enrichment unnecessarily. Enrichment is for unknown barcode workflows only. Scan and receive actions must prevent accidental duplicate submissions using the established submitting/ref pattern.

Core Inventory Pages

Inventory / Search Inventory

Display warehouse-scoped or all-property inventory state; search/filter supplies and parts; surface low-stock state. Warehouse context affects query keys. All-properties mode must not be used as mutation target. SignalR supply.updated should invalidate visible inventory data.

Parts Catalog

Master catalog of parts regardless of whether stocked at selected warehouse; part detail review; create/edit where authorized; identifier and vendor cross-reference subflows; label printing where implemented. Do not confuse catalog existence with warehouse supply existence.

Transactions

Read-only view of inventory ledger activity. Transactions are append-only from the API perspective — the PWA should not offer edit/delete transaction actions unless the API contract changes. supply.updated often implies transaction queries should be invalidated.

Orders

Active order workflows: creating or marking items as ordered, marking received, stock-in and receive combined workflow, canceling with reason, viewing active orders and history. Receiving should be tied to physical receipt of stock. Quantity confirmation must be explicit.

Manage pages are gated to DatabaseManager users in the PWA and protected by the API. UI gating is not enforcement.

Common Manage Page Pattern

Manage Parts

Administrative part maintenance, identifier management, vendor cross-reference management, and part deletion where API permits. Identifier uniqueness, ATC master rules, and similar-name validation are all API-owned.

Manage Warehouses

Warehouse/property record maintenance and PropCode management. Changing PropCode can affect {propCode}_wh role mapping and SignalR grouping assumptions. Warehouse changes should invalidate warehouse reference data.

Manage Invoice Aliases

Maintain vendor alias mappings used by invoice reconciliation. Alias changes affect future invoice matching. Do not silently remap ambiguous invoice descriptions without clear user confirmation.

API Client Layer

Standard API Fetch Behavior

The API fetch helper isolates transport details from page components. Expected responsibilities: read VITE_API_BASE_URL, acquire MSAL API token, attach bearer token, set JSON headers, serialize request body, parse JSON response, and convert error responses to a consistent frontend error shape.

⚠️
Page components should call service functions or hooks, not raw fetch, unless there is a deliberate exception.

Functions Client Rules

Keep VITE_FUNCTIONS_BASE_URL separate from VITE_API_BASE_URL
Do not call Functions for API-owned business rules
Handle Functions failures separately from API failures — response shapes may differ

Error Handling

API client errors must preserve enough information for page code to distinguish: unauthenticated (401), forbidden (403), not found (404), duplicate/conflict (409), validation failure, network failure, and unexpected server error.

⚠️
Do not reduce all errors to "Something went wrong" at the service layer. Page code needs meaningful categories to show useful remediation text.

Implementation Notes and Follow-Ups

VITE_REDIRECT_URI

Environment files include VITE_REDIRECT_URI, but current MSAL behavior derives redirect URI from the app origin and Vite base URL rather than directly consuming VITE_REDIRECT_URI. If a future developer changes msalConfig.ts to use it, update this section and retest redirect-loop hardening behavior.

Static Web App Config Verification

During final review, verify the exact staticwebapp.config.json used by Azure Static Web Apps. If more than one exists, reconcile them.

version.json Must Stay Network-Fresh

The app update flow depends on fetching fresh version metadata. Do not precache version.json or configure long-lived caching for it.

Role Checks Are UX Only

Any documentation or code comments that imply the frontend enforces authorization should be corrected. The API enforces all authorization rules server-side.
