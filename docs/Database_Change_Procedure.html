Database Change Procedure

How to make schema changes that apply correctly to both local SQLite and Azure SQL

Status Active
Created April 7, 2026
Scope API Repo
Applies To All Schema Changes
01 Overview 02 Why Two Steps 03 Procedure 04 Type Reference 05 How Production Updates
01

Overview

The API uses two different database providers depending on environment:

ℹ️
Local Development → SQLite (dev_data/inventory_dev.db)
Azure (Production) → Azure SQL (atc-inventory-db-dev via Key Vault connection string)

Because SQLite and SQL Server use different type systems, a single EF Core migration generated for one provider will fail on the other. The procedure below ensures both environments stay in sync without losing local test data.

02

Why Two Steps Are Required

EF Core migrations are generated against a specific database provider. When you run dotnet ef migrations add, the tooling defaults to the Development environment which loads appsettings.Development.json and resolves to SQLite. However, the model snapshot in source control is SQL Server — causing a type mismatch that prevents the migration from applying to either database cleanly.

The solution is to always generate migrations explicitly against the Production provider (SQL Server), then manually apply the equivalent SQLite changes to your local database.

⚠️
Never run dotnet ef migrations add without the ASPNETCORE_ENVIRONMENT=Production prefix. Doing so will generate a SQLite migration that corrupts the snapshot and causes cascading errors.
03

Procedure

Follow these steps every time you make a change that affects the database schema (new columns, new tables, modified constraints, etc.).

1
Make your entity changes in code
Add or modify properties on your entity classes in Entities/. Update any relevant DTOs.
2
Generate the Production migration
Use the ASPNETCORE_ENVIRONMENT=Production prefix to force SQL Server type generation. This creates the migration file but does not touch any database.
bash
ASPNETCORE_ENVIRONMENT=Production dotnet ef migrations add <MigrationName>
Verify the generated Up() method contains only the expected changes in SQL Server types (nvarchar, int, datetime2). If it contains TEXT or INTEGER types, something went wrong — run dotnet ef migrations remove and investigate.
3
Apply equivalent changes to local SQLite manually
Open the local SQLite database and apply the equivalent schema changes using SQLite-compatible types (see the Type Reference in §04). Then insert a record into __EFMigrationsHistory so EF knows this migration has been applied.
bash — open sqlite
sqlite3 dev_data/inventory_dev.db
sql — apply changes + record migration
-- Apply your schema changes using SQLite types
ALTER TABLE "TableName" ADD "ColumnName" TEXT NULL;
ALTER TABLE "TableName" ADD "OtherId" INTEGER NULL;

-- Mark the migration as applied (use the exact timestamp prefix from the migration filename)
INSERT INTO "__EFMigrationsHistory" ("MigrationId", "ProductVersion")
VALUES ('20260407140513_YourMigrationName', '10.0.5');

.quit
The MigrationId value must exactly match the migration filename without the .cs extension.
4
Test locally
Run dotnet run. The app should start cleanly. EF will see both migrations as already applied in __EFMigrationsHistory and skip them. Your existing test data is preserved.
bash
dotnet run
5
Push to GitHub
Commit and push your changes including the new migration file. The CI/CD pipeline builds, tests, and deploys the app. On restart, db.Database.Migrate() runs in the Production startup block and applies the migration to Azure SQL automatically.
bash
git add .
git commit -m "feat: description of your schema change"
git push
✅
No schema changes? If your push only affects endpoints, DTOs, frontend logic, or other non-schema code, skip this entire procedure and push normally. No migration needed.
04

Type Translation Reference

When writing the manual SQLite ALTER TABLE statements in Step 3, use this table to translate SQL Server types from the generated migration to their SQLite equivalents.

SQL Server Type SQLite Equivalent Notes
int INTEGER All integer sizes map to INTEGER in SQLite
nvarchar(max) TEXT Any nvarchar length maps to TEXT
nvarchar(450) TEXT Length constraints are not enforced in SQLite
datetime2 TEXT SQLite stores datetimes as ISO 8601 strings
bit INTEGER 0 = false, 1 = true
float REAL Floating point numbers
decimal(p,s) REAL SQLite does not enforce precision/scale
05

How Production Updates

The Azure SQL database is updated automatically on every deployment via db.Database.Migrate() in Program.cs:

if (app.Environment.IsProduction())
{
    using var scope = app.Services.CreateScope();
    var db = scope.ServiceProvider.GetRequiredService<InventoryDb>();
    db.Database.Migrate();
}

When the App Service restarts after a deployment, this block runs and applies any pending migrations against Azure SQL. EF checks the __EFMigrationsHistory table and only runs migrations that haven't been applied yet — making it safe to deploy repeatedly.

⚠️
Migration failure = app startup failure. If a migration throws an error against Azure SQL, the app will fail to start. Always verify your migration's Up() method is correct before pushing. In the event of a failed deployment, roll back the commit — the previous schema in Azure SQL is unaffected since the failed migration never completed.
ℹ️
Connection string location: The Azure SQL connection string is stored in Azure Key Vault as ConnectionStrings--DefaultConnection and is injected into the App Service at runtime. It is never stored in source control.
