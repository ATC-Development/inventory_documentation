F-10 Vendor Quote Request — Graph OBO Send Both Complete Implemented, tested, and production verified (May 2026). Replaces the original mailto: flow which was not viable in the ATC environment (New Outlook web wrapper, delayed/unreliable sends).

Architecture: API sends email on behalf of the logged-in user via Microsoft Graph On-Behalf-Of (OBO) flow. OBO is required (vs. app-only) so vendor replies route to the user's mailbox, not a shared mailbox.

API: POST /quote-requests/{token}/send. OBO exchange uses ATC-InventoryAPI-GraphOBO certificate from Key Vault. On success: sets Graph_sent_dt, Email_detected_dt, status → email_detected; creates QuoteRequestEmailEvent; fires SignalR orders.updated.

PWA: QuotePreflightPanel full rewrite — preview → create → compose (editable To/CC tag inputs) → send. Resend flow passes existingQuote prop to skip preview/create steps. Removed: buildMailtoUrl, ResendOverflowModal, clipboard fallback.

Retired: new_quote_email_detected PA flow turned off. QuoteEmailDetectedFunction removed from Functions repo. email-detected API endpoint retained (gated by FunctionAppQuote auth policy).

Schema additions: Graph_sent_dt, Graph_message_id, Graph_send_error on QuoteRequests table (migration 20260516130729_AddQuoteRequestGraphFields).

Config: QuoteTracking__TestMode — flip to false in App Service when ready to go fully live with real vendor addresses.
— Invoice Scan (F-01) Both Complete Implemented, tested, and production verified. Allows users to photograph or upload a vendor invoice; system extracts line items, matches to known parts, and writes stock-in transactions after user verification.

Pipeline: Image/PDF → AnalyzeInvoiceFunction (POST invoice/analyze) → ADI prebuilt-invoice model → VendorAliases lookup → GPT-4.1-mini matching for unresolved items → InvoiceVerificationModal → POST /supplies/scan/stockin batch + POST /vendor-aliases alias learning.

Azure resources: Azure Document Intelligence (S0, East US), Azure AI Foundry GPT-4.1-mini (Global Standard). 5 Key Vault secrets: AzureDocIntelligence--Endpoint/ApiKey, AzureOpenAI--Endpoint/ApiKey/DeploymentName.

Schema: VendorAliases table. Alias uniqueness for vendor-agnostic records (Vendor_id = NULL) enforced via API existence check — not a DB unique constraint.

PWA routes: /scan/invoice (ScanInvoicePage), /manage/invoice-aliases (ManageInvoiceAliases — DatabaseManager only).
— Equipment Tracking Both Complete Implemented, tested, and production verified. Tracks discrete tools and equipment across properties — separate domain from consumable inventory.

Schema: Equipment, EquipmentIdentifiers, EquipmentStatuses (current truth, upserted on movement), EquipmentTransactions (immutable ledger). TOOL_MASTER barcodes start at sequence 8000001; scan routing: value ≥ 8000000 → EquipmentIdentifiers, value < 8000000 → Identifiers.

API: Full CRUD (/equipment, /equipment/{id}), identifiers (/equipment/identifiers), scan operations (/equipment/scan/resolve, checkout, checkin, locationupdate), transactions (/equipment/transactions). SignalR equipment.changed broadcast on all writes.

PWA — Mobile: Tools section on home screen, /equipment/search (MobileEquipmentSearchPage with detail modal + in-place action phase machine), /equipment/scan (ScanToolPage — two-phase resolve then action).

PWA — Desktop: /equipment (EquipmentPage — table-first, filter by status/category/scope), /equipment/:id (EquipmentDetailPage — full record + transaction history + DM edit). Equipment management (create/edit/soft-delete) is handled through the main equipment pages — no separate /manage/equipment route.

Authorization: Create/edit/delete/permanent transfer = DatabaseManager. Checkout/checkin/location update = any authenticated user.

B-10 Shared Property Master Record Infra Blocked Both the Inventory app and CrisisApp maintain their own copies of property data (propCode, name, address, GPS coordinates). These will drift without a shared source of truth. B-12 makes this easier — once the notification channel is live, the DM can be automatically alerted to property changes rather than needing to manually reconcile.

Proposed architecture:
— SharePoint List as the master property record. Fields: PropCode, DisplayName, Address, Latitude, Longitude, YardiStatus, LastVerifiedDt.
— YARDI scheduled report → Power Automate flow → compare against SharePoint List → update if changed → notify DatabaseManager via F-20 for inventory app review.
— CrisisApp reads SharePoint List directly.
— Inventory app DM reviews and approves changes to Warehouses and ManagedProperties tables.

Blocked on: YARDI report format and delivery mechanism (email, blob, SFTP). Power Automate reconciliation flow design. Field mapping between YARDI columns and SharePoint List columns.
