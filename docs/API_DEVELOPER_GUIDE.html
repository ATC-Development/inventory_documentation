Purpose of This Guide

This guide is the primary developer reference for the ATC Inventory API. It is written for developers who need to maintain, troubleshoot, extend, or deploy the API without prior knowledge of the project history.

The API is the authoritative system of record for inventory data. It owns validation, authorization, business rules, database integrity, identifier resolution, inventory transactions, and server-side integration points. The frontend may guide users through workflows and Azure Functions may perform side effects, but the API is the only write authority for inventory state.

ℹ️
This guide covers: system context, local setup, startup pipeline, authentication, authorization, data model, business rules, endpoint inventory, SignalR, enrichment, Functions integration, testing, deployment, troubleshooting, and maintenance checklists.

This guide does not cover: per-endpoint request/response shapes, DTO field listings, or the full error code catalog. Those live in API_CONTRACT.html.

Source-of-Truth Rules

1Current API source code wins for implemented behavior. The API snapshot is the authority for mapped endpoints, route paths, middleware ordering, auth policies, and implemented DTO behavior.
2EF Core model and migrations win for actual database structure. SCHEMA.html documents the intended structure, but InventoryDb, entity classes, and migration files determine runtime behavior.
3Architecture and project rules win for design intent and responsibility boundaries. They explain why responsibilities are separated between the PWA, API, and Azure Functions.
4Frontend integration docs are contract references, not API internals. They explain how the PWA calls the API but are not a substitute for this developer guide.
⚠️
Known superseded reference — SignalR hub path: Older architecture documents reference the hub as /inventoryhub. The current implementation maps it at /hubs/inventory. This guide uses /hubs/inventory as canonical. Do not use /inventoryhub in any new code or documentation.

What the API Is

The authoritative inventory system of record
The owner of all write-side business rules
The validator of JWT bearer tokens issued by Entra ID
The enforcer of authorization policies (role-based and property-scoped)
The owner of the immutable transaction ledger
The host of the SignalR hub at /hubs/inventory
The owner of identifier resolution and enrichment workflow state

What the API Is Not

A UI layer
A background job scheduler or timer host
A place to trust frontend authorization decisions
A place to store passwords or local user accounts
A place for third-party provider API keys to be exposed to the client

System Request Flow

1 →User signs in to the React PWA using MSAL
2 →PWA requests an access token for the access_as_user API scope
3 →PWA calls API with Authorization: Bearer <access_token>
4 →API validates the token using Microsoft.Identity.Web
5 →API extracts identity and roles from JWT claims
6 →API enforces route-level and inline authorization checks
7 →API reads or writes via EF Core
8 →Successful writes append transaction records where applicable and emit SignalR invalidation events fire-and-forget

Understanding the startup order is required before modifying authentication, authorization, CORS, Swagger, SignalR, or endpoint mapping.

Service Registration (builder phase)

1 →Configure EF Core — SQLite for Development/Testing, SQL Server for Production
2 →Configure CORS policy (LocalFrontend) — includes local Vite ports and production Static Web App origin
3 →Add endpoint API explorer
4 →Configure HttpJsonOptions — register JsonStringEnumConverter so enums serialize as strings
5 →Configure authorization policies — DatabaseManager and FunctionApp
6 →Configure authentication — Microsoft.Identity.Web JWT bearer with EnableTokenAcquisitionToCallDownstreamApi
7 →Configure JwtBearerOptions — adds SignalR query-string token handler (reads ?access_token= for hub connections at /hubs/*)
8 →Configure Swagger / OpenAPI with OAuth2 PKCE definition
9 →Register scoped services: INameSimilarityService, INameValidationService, IChannelFailureReporter

14 →In Development: apply EF migrations, enable Swagger UI, map / redirect to /swagger
15 →In Production: apply EF migrations
16 →UseCors("LocalFrontend")
17 →UseAuthentication()
18 →UseAuthorization()
19 →Map all endpoint groups
20 →Map SignalR hub at /hubs/inventory — must be after UseAuthentication / UseAuthorization
21 →app.Run()

Authentication & Identity

Authentication Model

  • Every protected endpoint requires a valid bearer token
  • Identity is derived from token claims only — no local Users table
  • The frontend is responsible for acquiring the token; the API enforces authorization
http
Authorization: Bearer <access_token>

For SignalR hub connections, the PWA passes the token via query string because WebSocket negotiation cannot send Authorization headers:

websocket
wss://api-url/hubs/inventory?access_token=<token>

Claims Extracted by the API

The UserAuditHelper class extracts audit identity from token claims in a defined priority order.

ClaimRead From (priority order)
OIDoid → http://schemas.microsoft.com/identity/claims/objectidentifier
Email-like valuepreferred_username → upn → ClaimTypes.Upn → unique_name → email → ClaimTypes.Email → ClaimTypes.Name
Rolesroles claim in JWT; multiple values supported
⚠️
If the OID is missing on a write path, the helper throws an exception. OID is the stable user identifier for audit purposes. Email-like values may change if a user's UPN changes, but OID does not.

Diagnostic Identity Endpoint

GET /me — returns the token-derived OID, email, and current roles for the authenticated user. Use to debug claims and role assignments during development, confirm a new user has expected roles, and verify token content after a role change.

Authorization

Principles

The API is the enforcement boundary. Frontend role checks are UX conveniences only.
401 Unauthorized means the token is missing or invalid. 403 Forbidden means the token is valid but the user lacks the required permission.
DatabaseManager overrides all property-scoped restrictions.

Role Tiers

TierRole ClaimMeaning
Database ManagerDatabaseManagerFull access to all warehouses; all bulk, destructive, and administrative operations
Property Warehouse{propCode}_wh e.g. ms_whWarehouse write access for property-scoped operations on the matching warehouse
Baseline AuthenticatedValid token, no app roleAuthenticated reads, single-record creates, scan and adjustment workflows where the route permits
Function AppPropertyChannelFailure.WriteNotification failure logging endpoints — used by Azure Functions, not human users
UnauthenticatedNo valid tokenOnly routes that explicitly omit .RequireAuthorization()

Baseline Authenticated Access

Baseline authenticated access is not fully read-only. A baseline authenticated user can:

  • Read all inventory data where the route uses .RequireAuthorization()
  • Create single parts, single identifiers, single supplies
  • Perform scan consume and scan stock-in operations
  • Perform manual supply adjustments
  • Create inventory orders

A baseline authenticated user cannot: perform bulk creates, delete parts/supplies/identifiers, create/update/delete warehouses/categories/vendors, or access notification failure endpoints.

EF Core & Migrations

⚠️
Always generate migrations with ASPNETCORE_ENVIRONMENT=Production. Running without this prefix defaults to the Development environment (SQLite), generating SQLite types (TEXT, INTEGER) instead of SQL Server types (nvarchar, int, datetime2). This corrupts the EF model snapshot and causes cascading migration failures on both databases.

Full Migration Procedure

1
Update entity class and affected DTOs
Add or modify properties on entity classes in Entities/. Update any relevant DTOs.
2
Generate the migration with the Production prefix
Verify the generated Up() method contains only expected changes in SQL Server types. If it contains TEXT or INTEGER types, run dotnet ef migrations remove and repeat.
ASPNETCORE_ENVIRONMENT=Production dotnet ef migrations add <MigrationName>
3
Apply equivalent changes to local SQLite manually
Open the local SQLite database and apply equivalent schema changes using SQLite-compatible types. Then insert a record into __EFMigrationsHistory.
sqlite3 dev_data/inventory_dev.db

ALTER TABLE "TableName" ADD "ColumnName" TEXT NULL;

INSERT INTO "__EFMigrationsHistory" ("MigrationId", "ProductVersion")
VALUES ('20260407140513_YourMigrationName', '10.0.5');

.quit
4
Test locally
Run dotnet run. The API should start cleanly. EF sees the migration in __EFMigrationsHistory and skips it. Test data is preserved.
5
Push to GitHub
Commit and push. On production deployment, db.Database.Migrate() runs at startup and applies the migration to Azure SQL automatically.

High-Risk Migration Changes

Treat these with extra caution and verify against production data volume before deploying:

  • New unique constraints on populated tables
  • Required columns without defaults
  • Changes to identifier or supply uniqueness constraints
  • Any change to the transaction ledger structure
  • Changing FK delete behavior
  • Changing audit field names
⚠️
Migration failure = app startup failure. If a migration fails against Azure SQL, the app will not start. The schema in Azure SQL is unaffected since the failed migration never completed. Roll back the commit to recover.

Core Business Rules

Part Creation Rules

Part creation is atomic — if any component fails, the entire operation rolls back
Part name is required and must be unique; similar-name check runs before hitting the unique index (threshold: 0.85)
ATC_MASTER identifier is auto-generated by the server — never caller-supplied

ATC_MASTER Identifier Rules

Generated at part creation — never caller-supplied
Format: zero-padded 7-digit integer (e.g. 0000001). Always Issue_uom = EA and Unit_multiplier_to_base = 1 — hardcoded, never caller-controlled
POST /identifiers/create and POST /identifiers/create/multiple both reject caller-supplied ATC_MASTER with 400 ATC_MASTER_NOT_ALLOWED
A Part must have an ATC_MASTER identifier before it can receive supply — 422 PART_MISSING_ATC_MASTER if missing

Supply Rules

One supply row per part per warehouse — UNIQUE(Part_id, Warehouse_id)
Quantity is always stored in EA (base units)
Negative inventory is blocked on every quantity-changing path — this check must never be bypassed
Every quantity change automatically appends an immutable transaction row
POST /supplies/scan/stockin auto-creates the supply row if missing — triggers "New Part in Warehouse" transaction and DatabaseManager notification

Transaction Ledger Rules

Transactions are append-only — may not be created or deleted via public API routes
Every stock change records: Part/warehouse, quantity before/change/after, reason string and reason type, edited by OID and email, timestamp

Equipment Rules

Equipment is separate from parts/supplies — do not treat equipment as inventory parts
One physical item = one Equipment row; each has exactly one EquipmentStatus row (upserted on movement)
Equipment movements are recorded as EquipmentTransactions (immutable, append-only)
Create/edit/delete and permanent transfers require DatabaseManager. Checkout, checkin, and location update are open to any authenticated user.

Scan Rules

OperationEndpoint
Consume stock (scan out)POST /supplies/scan/consume
Receive stock (scan in)POST /supplies/scan/stockin
Unknown barcode (after 404)GET /identifiers/resolve/{code}

Purpose

SignalR is used for real-time cache invalidation only — not as a source of truth. The flow is: API writes to database → emits SignalR event fire-and-forget → PWA receives event → PWA invalidates React Query cache → PWA re-fetches authoritative state from API.

⚠️
SignalR event payloads are invalidation signals, not data payloads. The PWA must always re-fetch from the API. Hub broadcasts are fire-and-forget — if a broadcast fails, the inventory write has already succeeded. Log the failure; do not roll back.

Identifier Resolution & Enrichment

Resolve Endpoint Behavior

GET /identifiers/resolve/{code} — Auth: Authenticated

  • Known code: Returns the matched part and identifier. Status: FOUND.
  • Unknown code: Creates an IdentifierLookup row, starts enrichment asynchronously, and returns Lookup_Started status with a lookup_id for polling.

GET /identifiers/lookup/{lookupId} — Auth: Anonymous (the lookupId is a GUID and not guessable)

Integration tests use WebApplicationFactory<Program> from Microsoft.AspNetCore.Mvc.Testing with a test SQLite database. Auth is injected using test-specific claim principals. As of the last snapshot: 116 integration tests, all passing.

Security Rules

Never store passwords or local user accounts
Never trust frontend authorization decisions — API enforces all rules server-side
Never expose third-party provider API keys to the PWA or in error responses
Never commit secrets to source control
Never log bearer tokens, connection strings, or Key Vault values
Use Entra OID as the stable identifier for durable audit fields — not email
Enforce DatabaseManager for all structural and destructive operations
Preserve identifier value global uniqueness on every write path
Block negative inventory in every quantity-changing path without exception
Preserve transaction immutability — no public delete or update routes on transactions
