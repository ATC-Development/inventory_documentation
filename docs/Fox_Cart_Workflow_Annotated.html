src/components/QuoteRequest/VendorOrderActionsModal.tsx · pingExtension()
When the modal reaches the "method" step and web_cart is one of the available order methods, the component runs pingExtension(). It posts a ping message to the extension via window.postMessage with a 2-second timeout. The result sets the extensionAvailable state flag, which controls whether the "Fill Vendor Cart" button is rendered at all.
VendorOrderActionsModal.tsx · pingExtension()
function pingExtension(timeoutMs = 2000): Promise<boolean> {
  return new Promise((resolve) => {
    const requestId = `ping-${Date.now()}`;
    const timer = setTimeout(() => {
      window.removeEventListener("message", handler);
      resolve(false);  // timeout → Fill Web Cart hidden
    }, timeoutMs);

    function handler(event: MessageEvent) {
      if (event.data?.source !== "ATC_EXTENSION") return;
      if (event.data?.type !== "ping:response") return;
      if (event.data?.requestId !== requestId) return;
      resolve(event.data?.data?.type === "pong");
    }

    window.addEventListener("message", handler);
    window.postMessage({ source: "ATC_PWA", type: "ping", requestId }, window.location.origin);
  });
}
Why postMessage instead of chrome.runtime.sendMessage? The PWA page cannot call extension APIs directly. content-bridge.js (ISOLATED world, injected into the PWA origin) intercepts window.postMessage with source: "ATC_PWA" and relays it to the extension background via chrome.runtime.sendMessage.

The router checks the message type and dispatches to handlePing(), which validates the sender origin against APPROVED_PWA_ORIGINS before responding. The response travels back through content-bridge.js which posts it back to the PWA page.

content-bridge.js forwards check_login to background via chrome.runtime.sendMessage
src/content-bridge.js
content-bridge.js runs in the ISOLATED world on the PWA origin. It listens to window.postMessage events from the same window, validates source === "ATC_PWA", checks the type against an allowlist (ping, fill_cart, check_login), and relays the message to the background service worker.
src/content-bridge.js — outbound relay
const ALLOWED_TYPES = new Set(['ping', 'fill_cart', 'check_login']);

window.addEventListener('message', async (event) => {
  if (event.source !== window) return;
  if (event.origin !== PWA_ORIGIN) return;
  if (data.source !== 'ATC_PWA') return;
  if (!ALLOWED_TYPES.has(data.type)) return;

  const response = await chrome.runtime.sendMessage({
    type: data.type,
    requestId,
    correlationId,
    vendorKey,
    payload: data.payload ?? null,
  });

  // Echo the response back to the PWA page
  window.postMessage({
    source: 'ATC_EXTENSION',
    requestId,
    type: `${data.type}:response`,  // e.g. "check_login:response"
    ok: true,
    data: response,
  }, PWA_ORIGIN);
});

FOX
content.isolated.js runs handleCheckLogin() — requests kd values from MAIN world
src/vendors/fox/content.isolated.js · handleCheckLogin()
The fox isolated script receives the check_login message from the background. It calls requestKdValues(), which posts a fox_request_kd message to the MAIN world. Because the isolated world cannot access page JavaScript variables directly, it asks the MAIN world script to read them.
Why two content scripts? The Kodaris kd global is a constant declared in inline page JavaScript — it is not attached to window. Only the MAIN world script can read it. However, the MAIN world is accessible to page scripts, so all fetch calls and security logic must live in the ISOLATED world where the page cannot intercept them. The two scripts communicate via window.postMessage.
src/vendors/fox/content.main.js — reads kd and responds

  isLoggedIn = kd?.customer?.isLoggedIn ?? false;

src/vendors/fox/content.isolated.js · handleCheckLogin()
async function handleCheckLogin() {
  const kd = await requestKdValues();

  if (kd.isLoggedIn && Number(kd.companyId) === Number(COMPANY_ID)) {
    return { status: 'login_verified' };  // ✓ correct ATC account
  }

  // Not logged in or wrong account — store pending flag, show banner
  await chrome.runtime.sendMessage({
    type: 'set_overlay_state',
    key: PENDING_LOGIN_KEY,
    value: { vendorKey: 'fox' },
  });

  showLoginOverlay(kd.isLoggedIn && Number(kd.companyId) !== Number(COMPANY_ID));

  return { status: 'login_required' };
}
Identity check: COMPANY_ID = 2771 (from vendor.config.js). isLoggedIn && companyId === 2771 = ATC account confirmed. Wrong companyId = different user logged in — overlay shows a "wrong account" warning. Not logged in = show login prompt.

content.isolated.js · checkPendingLogin() (simplified)
async function checkPendingLogin() {
  const response = await chrome.runtime.sendMessage({
    type: 'get_overlay_state', key: PENDING_LOGIN_KEY,
  });
  const pending = response?.state ?? null;
  if (!pending) return;  // no pending login — nothing to do

  const kd = await requestKdValues();
  if (!kd.isLoggedIn || Number(kd.companyId) !== Number(COMPANY_ID)) return;

  // Logged in with correct account — clear flag, notify background
  await chrome.runtime.sendMessage({
    type: 'set_overlay_state', key: PENDING_LOGIN_KEY, value: null,
  });
  await chrome.runtime.sendMessage({ type: 'login_ready' });
}

src/api/vendorCart.ts · cartPreflight()
export async function cartPreflight(
  vendorId: number,
  warehouseId: number,
  orderIds: number[]
): Promise<CartPreflightResponse> {
  return apiFetch<CartPreflightResponse>(`/vendors/${vendorId}/cart/preflight`, {
    method: 'POST',
    body: JSON.stringify({
      warehouse_id: warehouseId,
      order_ids: orderIds,
    }),
  });
}
409 CART_SESSION_OPEN: If another manager already has an open session for this vendor, the API returns 409 with a CartCollisionError body (manager name, property, started time, can_override flag). The PWA shows the collision warning UI. DatabaseManager sees an "Override Session" button; regular managers see only "Save for Later" and "Cancel."
API
API validates orders, creates VendorCartSession, returns callback_token + item payload
inventory_api · POST /vendors/{vendorId}/cart/preflight
The API validates that all orders belong to the requesting user's warehouse, verifies they are in pending status, resolves VendorParts.Vendor_stock_number for each item, creates a VendorCartSessions record with status open, generates a GUID callback_token, and returns the response. The token expires at Expires_dt (60 minutes from creation).
What the callback_token is for: The isolated script on foxparts.com will later POST to /vendors/cart/confirm using this token — but it cannot use MSAL authentication (it's running inside a vendor website). The token is the authorization mechanism for that specific confirm call. It is scoped to one session, expires in 60 minutes, and is rate-limited to 5 consecutive failures.

content.isolated.js · getCart() — reads existing cart state
async function getCart(csrfToken) {
  // Fox API: POST [] to bulk endpoint returns current cart without modifying it
  const response = await fetch(CART_ENDPOINT, {
    method: 'POST',
    credentials: 'include',   // uses Fox session cookies (HttpOnly)
    headers: {
      'Content-Type': 'application/json',
      'X-CSRF-TOKEN': csrfToken,  // from kd.csrfToken (MAIN world)
    },
    body: JSON.stringify([]),
  });
  if (!response.ok) return null;
  return response.json();
}

export async function cartConfirm(
  sessionId: number,
  callbackToken: string,
  vendorOrderId: string | null,
  itemsAdded: number,
  itemsFailed: string[]
): Promise<void> {
  return apiFetch<void>('/vendors/cart/confirm', {
    method: 'POST',
    body: JSON.stringify({
      session_id: sessionId,
      callback_token: callbackToken,   // never logged — security requirement
      vendor_order_id: vendorOrderId,
      items_added: itemsAdded,
      items_failed: itemsFailed,
    }),
  });
}

ExtensionCartProvider.tsx · tryAbandon()
Once a preflight has created a VendorCartSession record, any failure must abandon it so the session slot is freed. Three conditions trigger tryAbandon():
— cart_fill_failed push received from extension
— cartConfirm() throws
— clearSession() called when preflight !== null and status is not "success" (user closes the panel early)

The abandonAttempted flag on the session prevents duplicate calls.
ExtensionCartProvider.tsx · tryAbandon() guard
const tryAbandon = useCallback(async (session: ActiveCartSession) => {
  if (
    session.preflight === null ||       // no API session created yet
    session.status === "success" ||     // already confirmed — don't abandon
    session.abandonAttempted            // guard against double-call
  ) return;

  setActiveSession((prev) => prev ? { ...prev, abandonAttempted: true } : prev);

  try {
    await cartAbandonSession(session.preflight.session_id);
  } catch {
    // Abandon is best-effort — do not surface to user
  }
}, []);

Background session — one at a time

The background service worker holds a single activeSession variable. If a second check_login or fill_cart arrives while a session is active, the background immediately responds with { success: false, error: 'Another session is already in progress.' }. This is separate from the API-level collision guard (VendorCartSessions soft lock per vendor), which blocks concurrent sessions for the same vendor even across different browsers or machines.
