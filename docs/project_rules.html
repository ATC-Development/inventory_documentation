Architecture Overview

The Inventory Application consists of three separate application repositories, each with its own GitHub repository, deployment pipeline, and responsibility boundary.

⚠️
These applications must not be combined into a single repository. Azure SignalR Service is a managed infrastructure component — not a fourth repo. It is configured through the Azure portal and consumed by the API.
Repo 1
inventory_pwa
React PWA Front End — Desktop + Mobile UX, MSAL authentication, token acquisition
Repo 2
inventory_api
ASP.NET Backend — System of record, business rules, CRUD + integrations, SignalR hub
Repo 3
inventory_functions
Azure Functions — Automations, side effects, reports, notifications, invoice analysis
02

Repo 1 — React PWA Front End

Purpose

Provide a single React application supporting desktop-first and mobile-first PWA UX
Handle MSAL authentication and acquire access tokens
Call the ASP.NET backend with Authorization: Bearer <token>

Rules

All authentication happens in the client using MSAL
The frontend must never enforce authorization as the source of truth. The API enforces permissions — a 403 response is always possible and must be handled gracefully
One React app must support both desktop and mobile layouts using responsive design and conditional rendering — not separate apps
SignalR connection is additive and non-blocking — the app must function fully if the hub is unavailable. No feature may fail or hang due to a missing hub connection
SignalR events are used exclusively to trigger React Query cache invalidation. Event payloads must never be used to drive UI state directly

UX Expectations

Mobile: scan-first, card-based, minimal typing
Desktop: table-first, keyboard/scanner friendly

Versioning

Semantic versioning (major.minor.patch)
Current version must be visible in the UI (splash screen or About page)
03

Repo 2 — ASP.NET Backend (System of Record)

Purpose

Act as the system of record for all inventory data
Validate MSAL access tokens and enforce authorization and business rules
Host the SignalR hub and broadcast real-time events to connected clients
Enqueue notification side-effect messages to inventory-notifications Azure Storage Queue

Authentication & Identity

JWT access tokens validated on every request via Microsoft.Identity.Web
Identity derived from token claims only — no local Users table
Audit fields recorded on all writes: *_by_oid (required), *_by_email (optional), timestamps

Authorization

RoleAccess
DatabaseManagerFull access — all properties, all admin and bulk operations
propCode_wh (e.g. ms_wh)Property staff — read all, write only to their assigned warehouse
No roleRead-only access to inventory data
DatabaseManager overrides all property role checks — can write to any warehouse regardless of group membership
The single-create / bulk-create split is deliberate: single creates are open to any authenticated user; bulk/destructive operations require DatabaseManager
PUT /parts/{id} is open to any authenticated user — part updates are a natural extension of the single-create permission
DatabaseManager is an M365/Entra ID group managed in the Azure portal — no code changes required to grant or revoke access
A 403 on a DatabaseManager-gated endpoint means the user is not in the M365 group — nothing is wrong with the application

SignalR

Hub endpoint: /hubs/inventory — registered in the API repo at startup
All write endpoints broadcast a real-time event after a successful response — fire-and-forget, never blocking
Azure SignalR Service connection string must be stored in Key Vault or App Service config — never in source control
EventScopeTrigger
supply.updatedWarehouse groupAny supply write — consume, stock-in, adjust, create, batch
supply.low_stockWarehouse groupSupply write where quantity_after ≤ min_level
part.createdAll clientsSingle part created or bulk batch complete
part.updatedAll clientsPart updated or deleted
warehouses.changedAll clientsWarehouse create, update, or delete
categories.changedAll clientsCategory create, update, or delete
vendors.changedAll clientsVendor create, update, or delete
equipment.changedAll clientsAny equipment write
orders.updatedAll clientsOrder status change, create, or Graph OBO send

Scan Flow

Scan Out: POST /supplies/scan/consume — decrements by qty_each EA directly, no multiplier
Scan In: POST /supplies/scan/stockin — applies multiplier, auto-creates supply row if missing
/identifiers/resolve/{code} is not part of normal scan operations — only triggered on 404 IDENTIFIER_NOT_FOUND
Auto-creating a supply row writes Reason = "New Part in Warehouse" — triggers Azure Functions alert to DatabaseManager

ATC_MASTER Barcode Rule

Every Part must have exactly one ATC_MASTER identifier — auto-generated by the server at Part creation, never caller-supplied
Any request including ATC_MASTER in the identifiers list is rejected with 400 ATC_MASTER_NOT_ALLOWED
Format: 7-digit zero-padded integer (e.g. 0000001). Unit_multiplier_to_base = 1 and Issue_uom = EA are hardcoded — never caller-controlled
ATC_MASTER values must remain below 8000000 — design boundary to avoid collision with TOOL_MASTER namespace

TOOL_MASTER Barcode Rule

Every Equipment item must have exactly one TOOL_MASTER identifier — auto-generated at Equipment creation, never caller-supplied
Any request including TOOL_MASTER is rejected with 400 TOOL_MASTER_NOT_ALLOWED. Cannot be added or deleted manually.
Stored value: bare integer (e.g. 8000001). Display format 8-000001 is UI-only — never stored or scanned

Barcode Namespace Routing

0000001 – 7999999
Parts / ATC_MASTER
Routes to Identifiers table. Any scanned value below 8000000.
8000001 +
Equipment / TOOL_MASTER
Routes to EquipmentIdentifiers table. Any 7-digit value ≥ 8000000.
ℹ️
On equipment scan screens, the PWA passes "tool" mode to normalizeBarcode(input, mode). In tool mode, pure numeric input is treated as an equipment barcode suffix — the app automatically prepends 8 and zero-pads to 7 digits. Example: user types 12 → resolves as 8000012. This shorthand is only active on equipment scan screens.

Supplies Base-Unit Rule

The Supplies table always represents EA-level (base unit) quantities
Unit_multiplier_to_base is a stock-in conversion only — applied at scan time, never persisted

Database Rules

EF Core with SQLite in development (dev_data/inventory_dev.db); Azure SQL in production
Unique constraints: Category.Name, Part.Name, Vendor.Name, Identifier.Id_value, Supply(Part_id, Warehouse_id), VendorParts(Vendor_id, Part_id), Warehouse(PropCode, Name)
Name normalization + similarity checks (threshold 0.85) prevent near-duplicate names before hitting UNIQUE indexes
Migrations auto-apply at startup in both Development and Production via db.Database.Migrate()
04

Repo 3 — Azure Functions (Automations)

Purpose

Azure Functions handle side effects and automation — not core CRUD logic.

Scheduled reports (daily/weekly/monthly)
Low-stock scans and notifications
PDF generation and label printing
Invoice analysis via Azure Document Intelligence + GPT-4.1-mini
Queue-triggered notification delivery via inventory-notifications

Rules

Functions must not expose CRUD endpoints that bypass API business rules. HTTP-triggered Functions (reports, labels, invoice analysis) are permitted — they do not write to the database
Functions must never write directly to the database — only read-only access via Managed Identity for report generation
Functions may call ASP.NET endpoints for operations requiring business rule enforcement
Functions may consume messages from inventory-notifications Azure Storage Queue via QueueTrigger
Secrets must not be stored in code — use Managed Identity or secure app credentials
Queue-based notification processing must not block user-facing API requests — the API writes to the queue fire-and-forget

Versioning

Automatic semantic versioning — version visible in Azure Portal

Cross-Cutting Rules

Separation of Concerns

LayerResponsibility
inventory_pwaUX, authentication, SignalR cache invalidation
inventory_apiValidation, authorization, business rules, integrations, SignalR hub
inventory_functionsAutomation and side effects

Security

Never store passwords
Never trust frontend authorization
Keep third-party API keys server-side only — never exposed to the frontend or in error responses
Never store Azure SignalR Service connection strings in source control — use Azure Key Vault or App Service configuration slots

CI/CD

Each repo deploys automatically from GitHub
Deployments must clearly identify: application, environment, and version
