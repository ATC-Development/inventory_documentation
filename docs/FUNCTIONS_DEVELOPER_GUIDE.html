01

Purpose & Scope

inventory_functions is the automation and side-effects layer of the ATC Inventory Application. It handles work that is intentionally kept out of the API: report generation, scheduled notifications, invoice analysis, and queue-based notification dispatch.

What This Repo Is Responsible For

  • Generating binary report output (Excel, PDF) on demand for the PWA
  • Analyzing vendor invoices via Azure Document Intelligence and Azure OpenAI
  • Sending scheduled Teams notifications via Power Automate flows
  • Dispatching queue-triggered notifications enqueued by the API

What This Repo Is Not

  • No inventory CRUD. Side-effect write-backs go through the API over HTTP — Functions never write to inventory, supply, order, transaction, or equipment tables directly
  • Not the authentication authority. JWT validation is performed locally in middleware, but identity is established by Microsoft Entra ID
  • Not responsible for SignalR broadcasts. Those are the API's responsibility
RepoResponsibility
inventory_pwaUI; calls Functions HTTP endpoints for reports and invoice analysis
inventory_apiSystem of record; enqueues notification messages; receives write-back calls from Functions
inventory_functionsSide effects; reads DB directly (read-only); calls API for write-backs

System Context

Two Call Patterns

Pattern 1
HTTP — Synchronous, User-Initiated
PWA sends authenticated request → Function validates JWT → reads DB or calls external AI services → returns binary file or JSON.
Used by: TabularReportFunction, LabelReportFunction, OrderFormReportFunction, AnalyzeInvoiceFunction
Pattern 2
Timer / Queue — Async, Self-Initiated
Timer fires or Storage Queue receives message → Function runs with no HTTP context → reads DB via Managed Identity → calls API for writes → POSTs to Power Automate for Teams delivery.
Used by: DailyPendingOrdersFunction, WeeklyOrderedItemsFunction, DailyChannelFailuresFunction, NotificationQueueFunction

Database Access Model

Both ReportDataService and NotificationDataService read the database directly using raw ADO.NET (DbCommand / DbConnection). EF Core is not used in this repo.

  • Development / Testing: SQLite via connection string pointing at inventory_api/dev_data/inventory_dev.db
  • Production: Azure SQL via SqlConnection with access token from DefaultAzureCredential (Managed Identity)
⚠️
All writes go through the API. Functions never write to inventory, supply, order, transaction, or equipment tables directly. Column names in GetOrdinal() calls must match the actual schema exactly — mismatches cause runtime exceptions.

Configuration Resolution Order

The codebase uses a three-path lookup in JwtValidationMiddleware, PowerAutomateService, and InvoiceAnalysisService:

csharp
_config[key]                         // root key (Azure App Settings / env vars)
?? _config[key.Replace("__", ":")]   // colon-normalized variant
?? _config[$"Values:{key}"]          // local.settings.json Values section

This means AzureAd__TenantId resolves as AzureAd:TenantId in ASP.NET Core config, or as Values:AzureAd__TenantId in local.settings.json. Key names do not need to change between environments.

Shared Dev Database

The Functions repo has no database of its own. The SQLite connection string must point to inventory_dev.db inside your local inventory_api checkout. The inventory_api project must be run at least once first so its startup migration creates the database file.

NotificationOverride

⚠️
Always enable NotificationOverride=true in local dev. When active, all Teams notifications are routed to the team/channel configured in NotificationOverride__TeamId and NotificationOverride__ChannelId — regardless of the intended property channel. If override IDs are not set, the service logs a Warning and falls back to the real channel (intentional — a misconfigured override does not silently drop notifications).

Authentication & Authorization

JwtValidationMiddleware

Located at Services/Auth/JwtValidationMiddleware.cs. This is the sole authentication enforcement point for HTTP triggers.

  • Timer and queue triggers: Middleware checks for HttpContext at the start of Invoke. When none exists, it calls next(context) immediately. No token required, negligible overhead.
  • HTTP triggers: Extracts Authorization: Bearer <token>, fetches signing keys from Entra ID OpenID Connect metadata, validates issuer/audience/lifetime/signing key. On success sets ITokenClaims on context.Features and sets httpContext.User. On failure returns 401.

Token Validation Failure Responses

ConditionStatusMessage
Missing/malformed Authorization header401Missing or malformed Authorization header.
Token expired401Token has expired.
Any other validation failure401Token validation failed.
AzureAd config incomplete500Server configuration error.

Accessing Claims in Function Handlers

csharp
var claims = functionContext.GetTokenClaims();
// claims.Oid   — object ID from token
// claims.Email — preferred_username or upn claim

If the middleware did not run (timer/queue), GetTokenClaims() returns an empty TokenClaims instance rather than null.

OID Cross-Validation

TabularReportFunction, LabelReportFunction, and OrderFormReportFunction enforce an additional check — the requested_by_oid field in the request body must match the OID from the validated token. This prevents one authenticated user from generating a report scoped to another user's identity.

csharp
if (!string.Equals(claims.Oid, request.RequestedByOid, StringComparison.OrdinalIgnoreCase))
    return new ObjectResult(new { error = "FORBIDDEN", message = "OID mismatch." })
        { StatusCode = StatusCodes.Status403Forbidden };
ℹ️
All HTTP triggers use AuthorizationLevel.Anonymous. This means the Azure Functions host does not enforce function keys. Authentication is enforced entirely by JwtValidationMiddleware. Do not change the authorization level without evaluating the middleware behavior.

Timer and Queue → API Call-Through

When timer and queue functions call the API for write-back operations, ApiClient acquires a token using DefaultAzureCredential. In production this resolves via the Function App's Managed Identity. Locally it falls through the credential chain (environment variables, Azure CLI, Visual Studio, etc.).

⚠️
This is not a user token — it is a service-to-service token scoped to the API's audience. The invoice analysis pipeline uses a different pattern: the user's own Bearer token is forwarded from the incoming HTTP request to the API calls within InvoiceAnalysisService.

Report Generation Pipeline

Data Access

ReportDataService reads the database directly using raw ADO.NET. Production connections use DefaultAzureCredential to acquire an access token for https://database.windows.net/.default and assign it to SqlConnection.AccessToken. The connection string itself contains no credentials.

Excel Generation

ExcelReportService uses ClosedXML to build .xlsx workbooks in memory and return byte[]. Column definitions, date format strings, and header styles are applied per report type.

PDF Generation

PdfReportService uses PdfSharpCore with CustomFontResolver registered as the global font resolver. The resolver's ResolveTypeface method falls back all unrecognized font family names to NotoSans. Even where report builders request "Arial" by name, PdfSharpCore resolves through the resolver and renders NotoSans. The embedded NotoSans-Regular.ttf is used for all PDF rendering in practice.

Barcode Rendering

ZXing.Net renders Code 128 barcodes as byte[] images, which are embedded in label PDFs by PdfReportService. Barcode values come from the Identifiers table via ReportDataService.

Report Type Filter Models

report_typeFilter Class
transactionsTransactionsFilters
part_trendPartTrendFilters
order_historyOrderHistoryFilters
on_orderOnOrderFilters
parts_catalogPartsCatalogFilters
cross_referenceCrossReferenceFilters

The TabularReportRequest.Filters field is a JsonElement deserialized into the appropriate filter class at dispatch time.

Binary Response Pattern

csharp
return new FileContentResult(bytes, "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet")
{
    FileDownloadName = $"order-form-{DateTime.UtcNow:yyyy-MM-dd}.xlsx"
};

The PWA receives this as a binary blob and triggers a browser download. The PWA sends report parameters (filters, IDs, format) — never pre-fetched row data.

Notification Pipeline

End-to-End Flow

1 →inventory_api write event → enqueues NotificationQueueMessage to inventory-notifications Storage Queue
2 →NotificationQueueFunction (queue trigger) → deserializes message → calls PowerAutomateService
3 →PowerAutomateService → resolves routing (override or real channel) → resolves PA flow URL from PA__{event_type} config → POSTs TeamNotificationPayload to PA flow
4a →On HTTP 2xx → calls ApiClient.PostNotificationSuccessAsync (fire-and-forget)
4b →On non-2xx or exception → calls ApiClient.PostNotificationFailureAsync (fire-and-forget)
5 →API records outcome in PropertyChannelFailureLogs or delivery success log

SanitizeForNotification

Located in Helpers/NotificationSanitizer.cs. Applied as a string extension on all user-sourced values before embedding in notification payloads. Power Automate's @{...} template substitution injects raw values directly into Adaptive Card JSON — unescaped double-quotes break the card's JSON structure.

csharp
public static string SanitizeForNotification(this string? value)
    => (value ?? string.Empty).Replace("\"", "in");

Replaces " with the literal text "in" at read time. Display-only — stored data is never modified.

DailyChannelFailuresFunction Atomicity

⚠️
Steps: (1) fetch unreported failures from DB, (2) dispatch digest to DM PA flow, (3) mark failures as reported via API. Step 3 is awaited (unlike fire-and-forget success/failure recording). If step 2 succeeds but step 3 fails, the failures will not be marked as reported and will reappear in the next day's digest. This is by design — it prefers duplicate reporting over silent data loss.

Invoice Analysis Pipeline

AnalyzeInvoiceFunction handles HTTP validation, form parsing, and file type checking, then delegates entirely to IInvoiceAnalysisService.AnalyzeAsync(file, warehouseId, vendorId, bearerToken).

Accepted File Types

JPEG, PNG, TIFF, application/pdf. Other content types return 400 UNSUPPORTED_FILE_TYPE.

Processing Stages

Stage 1 →Azure Document Intelligence — prebuilt-invoice model extracts VendorName, Items (Description, Quantity, Unit). If no items found, returns empty result immediately.
Stage 2 →Vendor alias lookup — normalized descriptions sent to POST /vendor-aliases/lookup. Matched items get match_source = "alias" and confidence = 1.0.
Stage 3 →Partition — items with alias match added to result immediately. Unresolved items proceed to Stage 4.
Stage 4 →Azure OpenAI semantic matching — parts catalog fetched once (GET /parts?limit=500). Each unresolved description sent to GPT deployment. Confidence ≥ 0.80 → match_source = "ai". Below threshold → match_source = "unresolved".

Match Source Values

match_sourceMeaning
aliasMatched by a known vendor alias in the database; confidence = 1.0
aiMatched by OpenAI semantic matching; confidence ≥ 0.80
unresolvedNot matched; requires manual review in the PWA

Bearer Token Passthrough

AnalyzeInvoiceFunction extracts the Bearer token from the incoming Authorization header and passes it to InvoiceAnalysisService.AnalyzeAsync. The service forwards this token to the API calls it makes (alias lookup, parts list fetch). This is a user-token passthrough — distinct from the Managed Identity token used by timer functions.

Key Vault Secret Resolution

InvoiceAnalysisService uses a GetSecret(localConfigKey, keyVaultSecretName) helper for ADI and OpenAI credentials. In local dev it reads from local.settings.json. In production it reads directly from Key Vault using DefaultAzureCredential. This is necessary because non-ConnectionStrings keys do not resolve through IConfiguration on Flex Consumption plans when set as Key Vault references.

Adding a New Function

Choosing the Trigger Type

ScenarioTrigger
User requests a file or analysis resultHTTP
Scheduled data aggregation or notificationTimer
Reactive to an API write eventQueue

HTTP Function Checklist

Declare AuthorizationLevel.Anonymous on HttpTrigger
Check claims.Oid immediately; return UnauthorizedResult() if empty
Apply OID cross-validation if request body contains user-scoped data (requested_by_oid)
Return structured JSON error objects (new { error = "CODE", message = "..." }) for all non-success responses
Inject services via constructor; do not call GetService directly
Use CancellationToken ct parameter and pass it through to all async calls

Timer Function Checklist

No HttpContext — no auth check needed
API calls use ApiClient (acquires Managed Identity token automatically)
Log start and completion with DateTime.UtcNow
Log myTimer.ScheduleStatus?.Next on completion
Catch exceptions per-item in loops; log and continue rather than aborting the entire run
Do not write to inventory, supply, order, transaction, or equipment tables directly

Queue Function Checklist

Deserialize defensively; on parse failure log the raw message and return — the message is consumed and discarded, not retried
Throw on dispatch exceptions (not deserialization failures) so the Functions host can retry via built-in retry policy
Log event_type and channel_type on dequeue for traceability

Adding a PA Flow

Add the flow URL key to local.settings.example.json under Values
Add the key to the paKeys array in Program.cs so it is validated at startup
Add a PA__{event_type} entry to Azure App Settings (or Key Vault) in production
Implement IPowerAutomateService dispatch calls for the new event type

Observability & Troubleshooting

Log Stream vs. Application Insights

  • Log stream (Azure portal → Function App → Log stream): Near-real-time Information-level and above. Use for watching timer runs, notification dispatch, and startup warnings as they happen.
  • Application Insights: Full exception detail including stack traces. Use outerMessage (not innermostMessage) in KQL queries. parsedStack array has fileName and method fields.

Common Failure Modes

SymptomLikely CauseWhere to Check

Known Cleanup Items

⚠️
NotificationQueueFunction misleading comment: The parse-failure path contains // don't throw — let the message be dead-lettered after retries. The actual behavior is the opposite: returning without throwing causes the message to be consumed and discarded immediately, with no retry and no dead-lettering. The comment should be corrected.
⚠️
AnalyzeInvoiceFunction diagnostic logs: Contains temporary diagnostic LogInformation calls that log whether the ADI endpoint and key config values are null or set. Should be removed before next production release.
⚠️
PowerAutomateService Console.WriteLine calls: DispatchAsync contains two Console.WriteLine calls that write the PA payload and response to stdout. These bypass structured logging and Application Insights — should be replaced with LogDebug or removed.
