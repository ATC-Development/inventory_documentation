Purpose

This document records how the PWA consumes backend APIs, Functions endpoints, query keys, SignalR events, and role-gated UI behavior. It answers PWA-maintenance questions such as:

  • Which PWA feature calls which backend endpoint?
  • Which query keys are owned by each feature?
  • Which SignalR event keeps a page fresh?
  • Which role gate controls a visible action?
  • What frontend error behavior should users see for common API failures?
ℹ️
This is not an endpoint contract. It does not replace API_CONTRACT.html. Keep implementation architecture, provider structure, MSAL bootstrap behavior, page patterns, testing strategy, and deployment instructions in PWA_DEVELOPER_GUIDE.html.

Authentication Rules

All protected API calls must use an access token acquired through MSAL for VITE_API_SCOPE.

Standard PWA API calls use apiFetch, which:

— reads VITE_API_BASE_URL
— gets the active MSAL account or sets the first available account active
— calls acquireTokenSilent
— attaches Authorization: Bearer <accessToken>
— sets Content-Type: application/json by default
— parses JSON responses where present
— throws ApiError for non-2xx responses
⚠️
Frontend role checks are UI-only. The API remains the final authorization authority. Every mutation path must handle 401 and 403 even when the UI hides unauthorized actions.

Functions calls may use VITE_FUNCTIONS_BASE_URL and a separate authenticated helper, especially when payloads are not JSON (e.g. invoice file upload).

Error Handling Expectations

The PWA should preserve enough error detail for pages to show useful remediation — not collapse every failure to a generic message.

Error CategoryTypical SourcePWA Expectation
401 Unauthorizedexpired/missing token, failed MSAL acquisitionPrompt re-authentication or route to login where appropriate; do not retry mutations blindly
403 ForbiddenAPI authorization policy rejects actionShow permission-focused message; keep UI stable; do not assume hidden buttons are sufficient
404 Not Foundstale row, deleted entity, wrong route/idShow not-found or refresh message; invalidate affected query if stale cache is likely
409 Conflictduplicate name, duplicate identifier, concurrency conflictShow conflict-specific message; in vendor alias confirmation, 409 may be non-fatal if alias already exists
Validation failureAPI model validation, missing required fieldsMap message to form-level or field-level feedback when available
Negative inventory / business-rule failurescan out or adjust would violate API rulesShow action-specific message; never force client-side override unless API explicitly supports it
Network failureoffline, API unavailable, Functions unavailableShow retry/connectivity message; preserve unsent form data where possible
SignalR unavailablehub disconnected, access token factory failureContinue normal API behavior; show real-time status only if useful; do not block workflows
Functions failureinvoice analysis/export/report generation failsDistinguish from API CRUD failure; show file/report-specific remediation

Feature-specific expectations:

  • Scan flows should preserve the scanned code and quantity after recoverable errors where safe
  • Forms should keep user-entered data when validation or network failures occur
  • Delete actions should require explicit confirmation and report 403, 409, or dependency failures clearly
  • Export/report failures should state whether the failure happened before request, during generation, or during download where that distinction is available

Maintenance Checklist

When adding or changing PWA backend usage, work through this checklist.

01Confirm the endpoint contract in API_CONTRACT.html or the current API source
02Add or update the PWA API service function — do not call raw fetch from page components unless there is a documented reason
03Add or update the route-to-data row in this file
04Add or update query key ownership and invalidation behavior
05Add or update SignalR event handling if the API emits a relevant event
06Add or update role/UI visibility expectations
07Handle 401, 403, 409, validation errors, network errors, and server errors intentionally
08Add tests for PWA behavior, especially cache invalidation and role-gated UI
09Do not duplicate full request/response contracts from API_CONTRACT.html
10If the change crosses into Functions, update the Functions usage matrix and verify VITE_FUNCTIONS_BASE_URL behavior
